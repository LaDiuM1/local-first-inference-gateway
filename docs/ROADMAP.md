# 추론 서버 구현 로드맵 (ROADMAP)

> 자체 추론 서버의 단계별 목표와 현재 완료 상태를 추적한다.
> 기술 선택의 판단 과정은 [`DECISIONS.md`](DECISIONS.md), 현재 외부 계약과 운영 절차는 [`API.md`](API.md)와 [`OPERATIONS.md`](OPERATIONS.md)에 둔다.
> 아래 수치는 각 단계에서 직접 확인한 실측값이며, 현재 자동화 테스트 수는 [README](../README.md)의 품질 현황을 기준으로 한다.

## 진행 원칙

1. **검증 뒤 확장**
   - 하드웨어 실행 가능성부터 확인하고, 각 단계의 계약과 실패 경로를 검증한 뒤 다음 책임을 추가한다.
2. **실측으로 결정**
   - GTX 1080 Ti 11GB 환경의 메모리, 지연, 처리량과 품질을 추측하지 않고 직접 측정한다.
3. **클라이언트와 내부 모델 분리**
   - 외부에는 지원 범위가 명시된 OpenAI 호환 계약만 제공하고, 실제 모델·업스트림·복구 정책은 게이트웨이가 소유한다.
4. **외부 일정과 저장소 완료 상태 분리**
   - 검색 모듈 E2E처럼 다른 팀 일정이 필요한 항목은 대기 상태를 명시하고, 저장소 안에서 완결되는 검증은 독립적으로 진행한다.

## 현재 아키텍처

```text
[openAt MSA 클라이언트]
      │  OpenAI 호환 HTTP · 서비스별 Bearer 키
      ▼
[Cloudflare 원격 관리형 Named Tunnel]
      ▼
[FastAPI 추론 게이트웨이]
      ├─ chat · vision → Ollama gemma4:12b-it-qat (GPU, 11434)
      ├─ embed        → Ollama snowflake-arctic-embed2 (CPU, 11435)
      └─ chat · vision의 로컬 실패만 → OpenAI gpt-5-mini
```

Ollama는 loopback에서만 실행한다. 게이트웨이는 인증, 요청 검증, 공개 별칭 라우팅, SSE 중계, 회로 차단기와 요청 관측을 담당한다. 임베딩은 벡터 공간 정합성을 위해 다른 모델로 폴백하지 않는다.

**기술 스택:** Python 3.12+ · uv · FastAPI · httpx(async) · pydantic-settings + YAML · Ollama · pytest · ruff

## 단계별 결과

### 1단계. 모델 검증 [완료 · 2026-07-08]

* 범용 모델은 `gemma4:12b-it-qat`, 임베딩은 `snowflake-arctic-embed2`로 확정했다.
* 생성 모델은 컨텍스트 8k 기준 VRAM 7.3GiB, 약 30.7 tok/s를 기록했다.
* 임베딩은 CPU 238ms, GPU 193ms로 단건 차이가 작았다. 범용 모델의 VRAM을 지키는 편이 전체 품질에 유리해 CPU 전용으로 분리했다.

### 2단계. 게이트웨이 기반 [완료 · 2026-07-09]

* OpenAI SDK가 Base URL만 바꿔 `/v1/chat/completions`를 호출할 수 있는 최소 FastAPI 게이트웨이를 구축했다.
* 클라이언트를 Ollama 규격에서 분리하기 위해 외부 계약은 OpenAI 형식으로 통일했다.

### 3단계. 스트리밍 [완료 · 2026-07-10]

* `stream: true` 응답을 전체 버퍼링하지 않고 SSE 이벤트 단위로 전달한다.
* 실 Ollama 검증에서 598개 델타가 첫 3.03초부터 마지막 25.29초까지 순차 수신되어 중계가 성립함을 확인했다.
* 깨진 JSON은 업스트림으로 보내지 않고 게이트웨이에서 OpenAI 형식 400으로 거절한다.

### 4단계. 라우팅과 임베딩 [완료 · 2026-07-13]

* 클라이언트는 `chat`, `vision`, `embed` 별칭만 사용하고 실제 모델·엔드포인트 매핑은 `routing.yaml`이 담당한다.
* `reasoning_effort` 생략은 일반 모드, 명시 값은 그대로 전달하며 잘못된 형식은 업스트림 호출 전에 거절한다.
* `/v1/embeddings`는 CPU 전용 Ollama를 호출하고 OpenAI 형식의 float/base64 응답과 입력 순서를 보존한다.

### 5단계. 폴백 [완료 · 2026-07-14]

* `chat`·`vision`의 로컬 연결 실패, 시간 초과, 429·5xx, 모델 부재와 응답 시작 전 무효 응답만 `gpt-5-mini`로 우회한다.
* 클라이언트 오류와 폴백 대상이 아닌 로컬 4xx는 그대로 반환한다. 응답 전달을 시작한 뒤에는 provider를 바꾸지 않는다.
* 별칭별 회로 차단기가 반복 실패를 빠르게 차단하고 half-open 단일 probe 성공으로 로컬 경로를 자동 복구한다.
* `embed`는 별도 프로세스로 격리하고 같은 모델만 재기동한다. 다른 provider의 벡터가 기존 색인에 섞이는 폴백은 허용하지 않는다.
* buffered·SSE 응답 검증, 압축 해제, 동적 hop-by-hop 헤더 제거, 취소·전송 장애 자원 정리를 계약 테스트로 고정했다.

### 6단계. 외부 공개와 인증 [완료 · 2026-07-15]

* 공유기 포트를 열지 않고 Cloudflare Named Tunnel이 loopback의 게이트웨이만 향하게 구성했다.
* `sk-oat-` 서비스별 키는 원본을 저장하지 않고 salt·SHA-256 digest와 식별 정보만 원자적으로 보관한다. 폐기는 재시작 없이 반영된다.
* `/v1/*`와 `/health`는 Bearer 인증 대상이고 `/docs`만 공개한다. 인증 실패는 본문 수신과 업스트림 호출 전에 끝난다.
* 현재 `/v1/*` 전체 요청 본문 상한은 Base64와 JSON 구조를 포함한 32MiB다.
* 로컬 응답 시작 기한은 요청 도착부터 90초, 폴백을 포함한 전체 응답 시작 기한은 115초다.
* 게이트웨이·두 Ollama·Watchdog은 독립 SYSTEM 작업, cloudflared는 공식 Windows 서비스로 운영한다. 실행 코드는 일반 사용자가 수정할 수 없는 관리자 전용 배포 사본으로 격리한다.
* 외부 모바일망 호출, 키 폐기, 실제 chat·streaming·vision·embed, 강제 종료 후 5.4초 복구와 재부팅 후 자동 기동을 실환경에서 확인했다.

### 7단계. 관측 [완료 · 2026-07-16]

* 모든 HTTP 요청은 종료 시점에 JSON Lines 한 줄로 기록한다.
* 도착·업스트림·응답 시작·완료 시간, 호출 주체, 공개 별칭, 최종 provider, 로컬 실패 사유, 상태·전송량·완결 여부를 남긴다.
* 프롬프트, 이미지, 응답 본문과 API 키 원문은 기록하지 않는다.
* 모든 응답의 `x-request-id`와 회로 차단기 상태 전환 기록으로 클라이언트 문의와 폴백 구간을 재구성할 수 있다.
* 전용 스레드가 16MiB × 5개 회전 파일에 기록해 요청 이벤트 루프를 막지 않는다.

### 8단계. MSA 클라이언트 연동 [게이트웨이 계약 완료 · E2E 대기]

* 첫 대상은 기존 OpenAI 규격을 사용하는 검색 모듈이다.
* `/v1/responses`의 `gpt-5.4-nano` 비스트리밍 이미지 요청은 내부 `vision` 정책으로 정규화한다.
* `text-embedding-3-small` 별칭은 로컬 1024차원 벡터를 zero-padding하여 외부 1536차원 계약을 맞춘다. 이 변환은 코사인 유사도를 보존하지만 provider 간 벡터 공간을 호환시키지는 않으므로 한 색인에 혼합하지 않는다.
* 전환 순서는 게이트웨이 재배포 → 검색 모듈 키 발급 → 클라이언트 전환 → 전체 재색인이다. 롤백할 때도 대상 provider 기준 전체 재색인이 필요하다.
* 게이트웨이 호환 계약과 검증은 완료했고, 검색 모듈의 실제 전환·E2E는 남아 있다.

### 9단계. 벤치마크 [완료 · 2026-07-17]

* 로컬은 외부 폴백을 끈 실제 게이트웨이 경로, 비교 모델은 OpenAI API 직접 호출로 측정했다.
* 생성 품질은 같은 문항·출력 상한과 각 모델의 저지연 일반 모드로 비교했다. 로컬 일반 모드는 고난도 16/18·블라인드 심판 8.89로 `gpt-5-mini` 14/18·7.67, `gpt-5.4-nano` 12/18·7.11보다 높았다.
* 로컬 임베딩은 동일 검색 표본에서 Recall@1 0.92·MRR 0.96으로 `text-embedding-3-small`의 0.88·0.91보다 높았다.
* 단건 첫 토큰 p95는 2.74초였고 동시 1·2·4·8건 전 구간에서 오류와 GPU 스필오버가 없었다. 다만 8건 첫 토큰 p95는 40.4초로 늘어 직렬 처리 대기열이 운영 한계임을 확인했다.
* 질의당 로컬 한계 전기비는 0.028원, 같은 프롬프트·출력 상한의 API 환산액은 `gpt-5-mini` 0.236원, `gpt-5.4-nano` 0.141원이었다.
* 상세 조건과 한계는 [`BENCHMARK_RESULTS.md`](BENCHMARK_RESULTS.md), 운영 해석은 [`BENCHMARK_FEEDBACK.md`](BENCHMARK_FEEDBACK.md)에 분리했다.

## 후속 운영 도구

* 요청 기록과 회로 이벤트를 읽고 구성 요소 상태를 함께 보여 주는 경량 모니터를 구현했다.
* 관리자 셸에서 필요할 때만 실행하며 상주 등록·외부 공개·알림은 하지 않는다. 자동 복구는 기존 Watchdog의 책임으로 유지한다.

## 범위 제외 (Non-Goals)

* Pascal GPU에서 공식 지원되지 않는 vLLM·TGI 등 별도 서빙 엔진 도입
* 단일 홈 서버를 넘어서는 멀티 노드 분산 처리와 Kubernetes 오토스케일링
* RAG와 프롬프트 계층을 넘어 모델 가중치를 직접 학습하는 파인튜닝
