# 추론 서버 구현 로드맵 (ROADMAP)

> 이 문서는 파이널 프로젝트의 자체 추론 서버 구축을 위한 전체 단계와 진행 현황을 추적한다.
> 각 구현 단계는 사전 정의된 검증 기준을 통과해야 완료로 간주한다.
> 세부적인 기술적 의사결정 이력은 [`DECISIONS.md`](DECISIONS.md)에 있다.

## 진행 원칙

1. **단계별 검증 (Step-by-step Verification)** 
   - 각 단계는 이전 단계의 안정성을 기반으로 진행되므로, 반드시 검증 스크립트(`scripts/verify_stageN.py`)를 통과한 후 다음으로 넘어간다.
2. **실측 기반 의사결정 (Data-Driven Decisions)** 
   - 현재 제한된 하드웨어(GTX 1080 Ti) 환경을 고려하여, 메모리 적재량, 추론 속도, 응답 품질 등은 모두 실제 측정값을 기준으로 판단한다.
3. **인터페이스 일관성 (Interface Consistency)** 
   - 외부 클라이언트의 구현 기술과 관계없이 접점 규격은 2단계에서 결정하며, 확정된 규격으로 통일하고 내부 모델 변경이나 처리 로직이 클라이언트에 누출되지 않도록 구조적으로 격리한다.

## 목표 아키텍처

```text
[openAt MSA 클라이언트 (사용 기술 독립)]
      │  추론 요청   ※ API 규격 미결정 (2단계)
      ▼
[외부 노출·인증 계층]  ※ 접근 방식·인증 방식 미결정 (6단계)
      ▼
[추론 게이트웨이 (FastAPI, 홈 PC)]  요청 중계   ※ 모델 요청 방식 미결정 (4단계) · 관측 범위 미결정 (7단계)
      ├─ 1차: 로컬 추론   ※ 서빙 엔진·모델 미결정
      └─ 폴백: 로컬 추론 장애 시 우회 경로   ※ 제공자·대상·조건 미결정 (5단계)
```

위 구성은 개념적 목표이며 구체 세부는 아직 확정하지 않았다. `※ 미결정`으로 표기한 부분은 표기된 예정 단계에서 결정한다.

**기술 스택:** Python 3.12+ / uv · FastAPI · httpx(async) · pydantic-settings · ruff

## 단계별 상세 진행 사항

### 1단계. 모델 결정 - 모델 검증 [예정]
현재 구축된 하드웨어 환경(VRAM 11GB)의 제약을 고려하여 최적의 범용 추론 모델 및 임베딩 모델을 결정하고, 실제 구동 환경에서 성능을 검증한다.

* **결정할 범위**
  * 모델 선정 — 하드웨어 한계 내에서 범용 추론·임베딩 각각에 사용할 모델을 무엇으로 할지
  * 자원 배치 — 제한된 VRAM에서 모델별로 GPU·CPU 중 어디에 적재할지
* **검증 기준**
  * [ ] 범용 추론 모델이 8k 컨텍스트(KV 캐시 포함)로 GPU에 적재되고 정상 응답하는지 확인
  * [ ] 임베딩 모델이 함께 상주 가능한 자원 배치에서 정상 응답하는지 확인
  * [ ] 동시 상주 시 VRAM 여유와 생성 속도를 실측으로 기록

### 2단계. 인프라 구축 - 게이트웨이 기반 구축 [예정]
이 단계에서 외부 클라이언트와의 접점 규격을 결정하고, 외부 클라이언트가 결정된 규격으로 추론 서버에 접근할 수 있도록 게이트웨이 API 인터페이스를 구축한다.

* **결정할 범위**
  * 게이트웨이 API 규격 — 클라이언트 접점을 어떤 규격으로 통일할지
  * 테스트 전략 — 서버 구현의 안정성을 어떤 테스트 범위로 보장할지
* **검증 기준**
  * [ ] 외부 클라이언트가 결정된 접점 규격으로 게이트웨이를 호출하고 정상 응답을 수신하는지 확인
  * [ ] 게이트웨이 골격(헬스체크·기본 추론 엔드포인트)이 계약 테스트를 통과하는지 확인

### 3단계. 핵심 기능 - 스트리밍 [예정]
사용자가 응답 대기 시간을 길게 느끼지 않도록(TTFT 최적화), 전체 문장을 버퍼링하지 않고 생성되는 토큰 단위로 즉각 중계한다.

* **결정할 범위**
  * 스트리밍 중계 방식 — 생성 토큰을 버퍼링 없이 순차 전달하는 구현
  * 요청 검증 책임 — 게이트웨이가 요청 형식을 어디까지 직접 검증할지
* **검증 기준**
  * [ ] 클라이언트에서 응답을 토큰 단위로 순차 수신하는지 확인
  * [ ] 형식이 잘못된 요청이 일관된 규격의 오류로 반환되는지 확인

### 4단계. 핵심 기능 - 모델 라우팅과 임베딩 [예정]
클라이언트가 모델을 어떤 형태로 요청할지와 실제 모델 선택 책임을 어디에 둘지 이 단계에서 결정하며, 검색을 위한 임베딩 처리 방식도 함께 다룬다.

* **결정할 범위**
  * 모델 요청 방식 — 클라이언트가 실제 모델명을 직접 지정할지 기능 추상화로 요청할지, 그리고 실제 모델 선택 책임을 어디에 둘지
  * 임베딩 처리 — 임베딩 요청을 게이트웨이 규격으로 어떻게 노출하고 어느 자원에서 실행할지
* **검증 기준**
  * [ ] 결정한 모델 요청 방식에 따라 대상 모델 호출과 전환이 정상 동작하는지 확인
  * [ ] 임베딩 API 호출 및 결과 반환 성공

### 5단계. 핵심 기능 - 폴백 [예정]
홈 서버 특성상 로컬 추론이 중단되면 클라이언트 요청 전체가 실패하는 단일 장애점(SPOF)이 된다. 로컬 추론 장애 시에도 서비스 연속성을 확보하는 것이 이 단계의 목표다.

* **결정할 범위**
  * 폴백 대상 — 어떤 제공자·모델로 우회하고, 어떤 기능은 우회에서 제외할지
  * 폴백 조건 — 무엇을 로컬 장애로 판정하고 어느 시점까지 우회를 허용할지
* **검증 기준**
  * [ ] 로컬 추론이 중단된 상태에서 폴백 범위 내 요청이 정상 응답으로 반환되는지 확인
  * [ ] 폴백 범위 밖의 요청이 잘못된 우회 없이 명시적 에러로 반환되는지 확인

### 6단계. 분석·평가 - 외부 노출과 인증 [예정]
추론 서버는 내부망 홈 PC에 있으므로, 홈 네트워크를 직접 노출하지 않으면서 외부 클라이언트가 접근할 경로와 인증 수단이 필요하다. 미인증 요청을 차단하고 재부팅 후에도 자동으로 복구되는 것이 이 단계의 목표다.

* **결정할 범위**
  * 외부 접근 방식 — 내부망 서버를 어떻게 외부에 노출하고 안정적인 접근 지점을 제공할지
  * 인증 방식 — 허용된 클라이언트만 통과시키기 위한 인증 수단과 미인증 요청 처리 규격
* **검증 기준**
  * [ ] 외부 네트워크 환경(예: 모바일 테더링)에서 추론 서버 호출에 성공하는지 확인
  * [ ] 인증 정보가 없거나 잘못된 요청이 업스트림 도달 전에 차단되는지 확인
  * [ ] 서버 재부팅 후 별도 조작 없이 전체 스택이 정상 복구되는지 확인

### 7단계. 분석·평가 - 관측 [예정]
병목 지점과 장애 원인을 사후에 진단하려면 요청 처리 과정이 관측 가능해야 한다. 이후 벤치마크의 근거가 될 운영 지표를 남기는 것이 이 단계의 목표다.

* **결정할 범위**
  * 관측 범위 — 병목·장애 진단을 위해 요청 처리의 어디까지 가시화할지
  * 수집 지표 — 지연·처리량·대기·폴백 발생 등 무엇을 어떤 형태로 남길지
* **검증 기준**
  * [ ] 동시 요청 상황에서 요청별 처리 경과와 대기·지연이 사후 진단 가능한 형태로 기록되는지 확인

### 8단계. 분석·평가 - MSA 클라이언트 연동 [예정]
자체 추론 서버가 openAt의 각 MSA 클라이언트에서 실제로 동작하는지는 최종 E2E 환경에서만 확인할 수 있다. 게이트웨이가 목표한 인터페이스 호환성을 실제 서비스로 입증하는 것이 이 단계의 목표다.

* **결정할 범위**
  * 연동 범위 — 실제 MSA 서비스의 어떤 AI 기능 흐름과 클라이언트 기술을 E2E 검증 대상으로 삼을지
  * 전환 방식 — 각 클라이언트를 자체 추론 서버로 전환하는 절차와 되돌릴 조건
* **검증 기준**
  * [ ] 실제 MSA 클라이언트가 자체 추론 서버를 호출하고 정상 응답을 처리하는지 확인

### 9단계. 분석·평가 - 벤치마크 [예정]
자체 추론 서버가 실제 서비스에서 가지는 성능·품질·비용의 효용을 정량 데이터로 확인해야 한다. 비교 대상과 평가 기준을 세우고 그 결과를 산출하는 것이 이 단계의 목표다.

* **결정할 범위**
  * 평가 기준 — 어떤 성능·품질·비용 지표로 효용을 측정하고 무엇을 합격선으로 볼지
  * 비교 대상과 질의 셋 — 무엇과 비교하고 어떤 질문으로 품질을 평가할지
* **검증 기준**
  * [ ] 결정된 기준에 따라 로컬·비교 대상 간 효용 비교 결과표가 산출되는지 확인
  * [ ] 벤치마크 결과를 바탕으로 운영·라우팅 정책의 후속 조정 근거가 도출되는지 확인

## 범위 제외 (Non-Goals)

다음 항목들은 현재 스프린트 및 하드웨어 한계 상 이번 프로젝트 범위에서 명시적으로 제외한다.

* **vLLM / TGI 등 별도 서빙 엔진 도입:** 현재 보유한 GPU(GTX 1080 Ti)는 Pascal 아키텍처(Compute Capability 6.1)로 최신 서빙 프레임워크 지원 대상에서 제외됨
* **멀티 노드 분산 처리 및 K8s 기반 오토스케일링:** 단일 로컬 홈 서버 환경이므로 컨테이너 오케스트레이션 배제
* **자체 모델 파인튜닝 (Fine-tuning):** RAG(검색 증강 생성) 프롬프트 엔지니어링으로 도메인 지식을 해결하며, 모델 가중치 훈련은 진행하지 않음
