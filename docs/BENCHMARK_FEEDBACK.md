# 자체 추론 서버 벤치마크 — 평가 및 운영 피드백

> - 평가 근거: [BENCHMARK_RESULTS.md](BENCHMARK_RESULTS.md)
> - 평가 범위: 성능·생성 품질·임베딩 검색 품질·비용·운영 제약

## 종합 평가

자체 추론 서버를 서비스의 기본 경로로 사용할 근거가 충분하다. 로컬 생성 모델은 품질 합격선을
충족한 것을 넘어 블라인드 심판 점수에서 두 비교 모델을 앞섰고, 단건 응답 속도와 동시 요청
안정성도 사전에 정한 기준을 통과하였다. 검색 임베딩 역시 현재 서비스 셋에서 OpenAI 모델보다
낮지 않은 결과를 기록하였다.

로컬을 선택한 핵심 근거는 비용만이 아니다. 생성과 검색 품질을 유지하면서 추론 경로와 모델 동작을
직접 제어할 수 있고, 외부 API는 장애와 처리 한계를 보완하는 역할로 제한할 수 있다. 품질 기준을
먼저 통과한 뒤 비용 효율을 확인했다는 점에서 로컬 기본 경로의 설계 명분이 분명하다.

## 잘된 점

### 1. 변별력 있는 문항으로 생성 품질을 비교하였다

공개 리더보드 점수를 인용하는 대신 KMMLU 6개 분야와 다단계 계산·논리·근거 함정·형식 준수로
구성한 자체 고난도 문항을 같은 문항·출력 상한과 각 모델의 저지연 일반 모드로 직접 측정하였다.
모든 모델이 고득점을 받는 쉬운 과제를 피했기 때문에 과제 유형별로 정답이 갈리는 지점에서 모델 간
차이를 관측할 수 있었다.

로컬 일반 모드는 고난도 16/18과 심판 점수 8.89로 두 비교 모델(14/18·7.67, 12/18·7.11)보다
앞섰고, 사고 모드는 계산·논리·근거 함정 전 문항을 맞혀(고난도 17/18) 기본 응답과 복합
요청의 역할을 분리할 근거도 확보하였다.

### 2. 로컬 응답과 외부 폴백을 측정 단계에서 분리하였다

로컬 게이트웨이의 외부 폴백을 비활성화하고 관측 기록의 `provider=local`을 확인하였다. 따라서
로컬 결과에 외부 모델 응답이 섞여 품질이나 성능이 높게 보일 가능성을 차단하였다. 실제 운영 경로인
게이트웨이를 통과시켜 측정한 점도 결과의 활용도를 높인다.

### 3. 정답 채점과 심판 채점을 역할에 맞게 사용하였다

KMMLU, 계산, 논리처럼 정답이 있는 문제는 자동 채점하고, 근거 충실도와 형식 품질처럼 하나의
정답으로 판단하기 어려운 문제만 블라인드 심판을 사용하였다. 생성 모델의 주관적 평가를 모든
문항에 적용하지 않아 품질 결론의 근거가 명확하다.

### 4. 생성과 검색을 하나의 로컬 경로로 묶을 근거를 확보하였다

로컬 임베딩은 Recall@1 22/24, Recall@3 24/24로 OpenAI 임베딩의 21/24, 23/24보다 각각 한
문항 앞섰다. CPU에서 임베딩을 처리하여 생성 모델의 GPU 메모리를 침범하지 않고, 게이트웨이에서
기존 1536차원 응답 계약도 유지한다. 제한된 하드웨어 안에서 생성과 검색의 책임을 분리한 구성이
실측으로 검증되었다.

## 운영 피드백

### 1. 동시성은 오류가 아니라 대기 시간으로 제어해야 한다

동시 8건까지 오류와 GPU 스필오버는 없었지만 총처리량은 약 16~18 tok/s로 일정했고 첫 토큰 p95는
40.38초까지 늘었다. 서버가 요청을 잃지 않는 것과 사용자가 기다릴 수 있는 시간 안에 응답하는 것은
다른 문제다.

현재 결과에서는 단건 첫 토큰 p95 2.74초만 5초 목표를 충족한다. 실제 운영에서는 활성 생성 수,
큐 길이, 예상 대기 시간을 관측하여 로컬에서 기다릴 요청과 외부 경로로 보낼 요청을 구분해야 한다.
다중 요청 대응의 다음 과제는 GPU 메모리 증설보다 대기열과 라우팅 정책이다.

### 2. 사고 모드는 정확도 향상이 지연보다 중요한 요청에 사용해야 한다

사고 모드는 계산·논리·근거 함정 전 문항을 맞혔지만 평균 응답 시간은 80.0초였고, 형식 준수
1문항은 정답 내용을 코드펜스로 감싸 출력 지시를 어겼다. 일반 대화에 항상 적용하면 체감 품질이
오히려 낮아질 수 있고, 형식이 엄격한 자동화 경로에서는 사고 모드 응답도 형식 검증을 거쳐야
한다. 계산, 복합 조건 분석, 비동기 작업처럼 긴 응답을 허용할 수 있는 요청에만 선택적으로
적용하는 것이 적절하다.

### 3. 전문지식 부족은 RAG가 담당해야 한다

Law 분야는 사고 모드에서도 정답 수가 늘지 않았다. 모델에 없는 지식은 더 오래 추론한다고 생기지
않는다. 반면 근거를 제공한 함정 과제에서는 로컬 모델이 전부 정답을 기록하였다. 전문 영역은 검색
모듈이 검증된 근거를 제공하고 생성 모델은 그 근거를 해석하도록 역할을 나누는 것이 맞다.

### 4. 임베딩 provider 변경은 전체 재색인과 함께 진행해야 한다

zero-padding은 로컬 벡터의 코사인 유사도를 보존하지만 OpenAI 벡터와 같은 공간으로 바꾸지는 않는다.
로컬 전환과 OpenAI 롤백 모두 해당 provider 기준의 전체 재색인이 필요하다. 서로 다른 벡터를 같은
색인에 섞지 않는 규칙을 실제 검색 모듈 E2E 검증에서도 유지해야 한다.

### 5. 비용 결과는 한계비용과 총비용을 구분해 사용해야 한다

로컬 0.028원과 API 0.141~0.236원의 차이는 이미 보유한 장비에서 요청 한 건을 추가로 처리하는
비용 비교다. 장비 구입비와 유휴 전력, 유지보수 시간은 포함하지 않았다. 이 결과는 품질을
유지하면서 추가 요청 처리 비용을 낮췄다는 근거로 사용하되, 총소유비용이 같은 비율로 낮아졌다고
해석하지 않는 것이 정확하다.

## 후속 확인 사항

| 우선순위 | 항목 | 완료 조건 |
|:---:|---|---|
| 높음 | 동시 요청 라우팅 기준 | 활성 생성 수·큐 대기 시간·외부 전환 조건을 운영 정책으로 확정 |
| 높음 | 검색 모듈 전환 | 로컬 임베딩 전체 재색인 후 실제 MSA 검색 흐름 E2E 통과 |
| 중간 | 사고 모드 적용 범위 | 계산·복합 분석 등 허용 요청과 응답 제한 시간을 클라이언트 계약으로 정리 |
| 중간 | 이미지 품질 평가 | 실제 서비스 이미지 유형을 반영한 품질 평가 후 전용 라우팅 필요성 판단 |
| 낮음 | 총소유비용 추적 | 실제 요청량이 쌓인 뒤 유휴 전력·운영 시간까지 포함한 월 단위 비용 산정 |

## 최종 피드백

자체 추론 서버는 제한된 하드웨어를 사용했다는 사실 자체보다, 제한 안에서 품질·성능·비용을
측정하고 외부 모델과의 책임을 나눈 과정이 강점이다. 로컬 일반 모드는 정상 요청의 기본 경로,
사고 모드는 복합 요청의 정확도 보완, 외부 API는 장애 대응, RAG는 전문지식 보완으로 각 역할이
분리되었다. 대기열 초과 시의 외부 전환은 위 라우팅 기준 확정과 함께 다음 운영 과제로 남는다.

이번 벤치마크는 “로컬 모델도 실행된다”는 수준을 넘어 실제 서비스에서 로컬을 우선 사용할 수 있는
조건과 사용하면 안 되는 조건을 함께 제시하였다. 이에 모델 선택 단계는 완료된 것으로 평가하며,
이후 작업은 검색 모듈 E2E 전환과 동시 요청 라우팅 정책을 운영 환경에서 검증하는 데 집중하는 것이
적절하다.
