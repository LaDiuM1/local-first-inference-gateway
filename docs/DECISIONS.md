# 추론 서버 결정 이력 (Decision Log)

> 파이널에서 담당하는 자체 추론 서버의 핵심 아키텍처와 기술적 결정 사항들을 기록해 두는 공간입니다.  
> 각 결정의 배경 및 판단 근거와 트레이드 오프를 작성함으로써 결정 이력 추적 및 향후 비효율적 결정이라는 판단이 섰을 시 피드백 작성을 통해 개발 사고 확장을 위한 목적으로 사용됩니다.
>
> **배경:** 팀 프로젝트(openat) 파이널에 RAG 등 AI 활용 기능이 추가되며, 그 AI 서비스가 호출할 LLM 추론 백엔드를 홈 GPU 서버(GTX 1080 Ti)에 직접 구축합니다. 로컬 모델(Ollama)을 1차로 사용하고 장애 시 대체 추론 경로를 통해 서비스 연속성을 확보하는 하이브리드 구조가 목표입니다.

## 2026-07-01

### 1. 파이널 프로젝트 담당 기능 결정
* **맥락 및 배경 (Context):** 파이널 프로젝트는 LLM을 활용한 기능 구현이 주목적이며 팀원별로 RAG 구현, 이미지 분석 구현 등 담당 기능을 결정하기 시작하였으며, 본인도 파이널 때 담당할 기능을 결정해야 한다.
* **결정 사항 (Decision):** LLM에 사용되는 추론 서버를 직접 구현하기로 결정하였다. 팀원 중 데브코스에서 지원하는 제한된 토큰 사용량에 대해 고민하는 팀원이 있었으며, 고민을 듣는 순간 이것을 해결해야 할 '문제'로 정의해야겠다고 생각하였다.
  이에 최근 오픈소스 모델(Ollama 서빙 프레임워크 이용 기준)도 성능이 많이 올라왔다는 말도 들어보았기 때문에 1차적으로 추론 서버 직접 구축이라는 아이디어를 떠올렸으며, 실제 리서치 해보니 꽤 괜찮은 성능을 보여주는 것으로 보여 내가 소유한 데스크탑의 GTX 1080 ti 기준의 그래픽카드로도 이 프로젝트 범위에서는 괜찮은 품질의 추론 성능의 실현 가능성이 존재한다고 판단하여 추론 서버를 직접 구현하는 것으로 결정하였다.
* **트레이드 오프 (Trade-off):** 실제 추론 서버 구현 시 하드웨어 제약으로 인해 품질이 떨어질 수 있는 문제가 발생할 수 있으나, 이 경우 무거운 모델은 openAi 모델로 반환, 검증된 허깅페이스 모델은 로컬 서버를 사용하는 것으로 생각중이다.
  특히 딥러닝 구축에 유리한 파이썬 기반 서버로 구축 예정인데 현재 파이썬은 기본 문법과 FastApi 기술의 키워드 정도만 아는 상태이다. 하지만 이 프로젝트의 궁극적인 목적은 백엔드 역량 증명이며, 이러한 제약 상황에서 높은 품질의 추론 서버 구축은 분명한 개발 엔지니어링 능력 증명에 경쟁력이 있을 것이라 생각한다.

상기 사유가 이 프로젝트를 생성하게 된 결정 및 배경이다.

## 2026-07-03

### 1. 로컬 LLM 서빙 프레임워크 결정
* **맥락 및 배경 (Context):** 최초 아이디어인 Ollama 서빙 프레임워크가 현재 하드웨어(1080 Ti) 환경과 프로젝트 목적상 최선인지 확인이 필요하며 vLLM, TGI 등의 선택지들과의 비교 후 결정이 필요하다.
* **결정 사항 (Decision):** Ollama 사용으로 결정하였으며, 이유는 보유 중인 1080 Ti의 파스칼 아키텍처를 vLLM, TGI 같은 최신 엔진들이 공식 지원하지 않는다. 더불어 VRAM이 프로젝트 요구 대비 한정적(11GB)이므로 GGUF 양자화로 메모리 최적화 및 CPU까지 활용할 수 있도록 도와주는 Ollama(llama.cpp 도구 활용) 사용이 합리적이라 할 수 있다.
* **트레이드 오프 (Trade-off):** 세부 파라미터 제어나 고도화된 동시 처리 최적화 세팅은 불리할 수 있다. 이는 다른 선택지 대비 불리한 요소긴 하지만 현재 환경에서는 유일한 선택지가 하나이므로, 향후 평가 단계에서 성능 미달 시 한정된 도구로 최적화가 필요할 수 있다. 

## 2026-07-08

### 1. [Sprint 1, 모델 결정] 추론 서버 하드웨어 환경에 최적화된 모델 결정

- **맥락 및 배경 (Context):** 현재 로컬 서버 하드웨어 환경에서 품질·추론 기능 요구를 고려하여 모델별 실측값 기반으로 최적의 모델 선정이 필요하다.
- **결정 사항 (Decision):** 검색 임베딩: `snowflake-ko`, 챗봇·이미지 분석: `gemma4:12b`를 사용하는 것으로 결정하였다.
  - **선정 기준**

    현재 프로젝트의 추론서버 요구는 챗봇, 이미지 분석 등의 범용 추론 모델과 검색 임베딩용 모델 두 가지이다. 이 기준으로 각각 모델을 선정하였으며 그 이유는 아래와 같다.
    - `gemma4:12b`: ollama의 범용 추론 모델은 현재 하드웨어 환경에서 사용 가능한 모델중 추론 능력 벤치마크 확인 시 `Gemma4 12B`가 압도적인 추론 지표를 보여줌을 확인했다. `DeepSeek R1 8B` 등 다른 모델도 있으나, 범용 추론 성능에서 큰 차이를 보이므로 1순위는 `gemma4:12b`를 사용하는 것으로 결정하였으며, 실측 시 양자화 모델로도 하드웨어(VRAM)이 감당 불가능할 때 `DeepSeek R1 8B`등의 모델도 고려하는 것으로 결정하였다.

      실측 결과 -`gemma4 12b 8k` 기준 VRAM 8.2GB, 유휴 1.5GB 총 11.248GB로 가용 메모리가 1.5GB로 임베딩 모델을 제외하고도 KV 캐시 리스크를 고려시 여유가 상당히 적음을 확인하였다, 이에 QAT 4B 양자화 모델로 변경하여 실측 시 가용 메모리가 약 400MB 증가하여 가용 VRAM이 1.9GB로 조금 더 여유가 확보되었으며, 추론 모델은 이 기준으로 결정 후 임베딩 전략 결정 후 필요 시 수정이 필요해보인다.
    - `snowflake-ko`: 추론 품질을 위해 가용 메모리를 범용 모델에 올인한 격이라 임베딩 가용 메모리가 사실상 없다. 이에 1차 아이디어는 임베딩은 추론 요구 성능이 비교적 낮으므로 GPU 대신 CPU 자원으로 위임하는 것을 생각하였다. 이에 현재 검색 담당자가 설정한 기술과 조건인 dance-vector, 한국어 기준의 특화 모델을 리서치하였고, 해당 기준의 리더보드 순위 기준 CPU(8700K 5.0Ghz)를 고려하였을 때 무거운 임베딩 모델(`pplx-embed-v1-4b`) 보다는 상대적으로 가볍지만 임베딩 성능으로도 충분한 `snowflake-ko`(리더보드 2위)를 사용하는 것으로 결정하였다.
    - 리더보드는 https://github.com/OnAnd0n/ko-embedding-leaderboard 를 참고하였다.

> **표기 정리:** 이 항목의 `snowflake-ko`는 모델을 검토할 때 사용한 후보 표기다. 리더보드에서 확인한 후보는 snowflake-arctic-embed 계열의 한국어 파인튜닝 파생판인데 Ollama 공식 라이브러리에는 원본 다국어 모델만 있어, 같은 1단계의 최초 실측 스크립트부터 실제 Ollama 식별자는 원본인 `snowflake-arctic-embed2`를 사용했으며 이후 코드·로드맵과 2026-07-14 결정 이력도 이 식별자를 기준으로 한다. 가벼운 임베딩을 CPU로 돌린다는 결정 취지와 아래 실측 판단 기준은 이 모델 그대로다.
- **트레이드 오프 (Trade-off):** 임베딩을 GPU -> CPU로 변경 시의 벤치마크를 각각 측정 시 내부 토큰 배치 처리량이 6.5배 느려지는 것을 확인하였다. (21.7 → 3.4 청크/s). 다만 임베딩 질의는 시나리오상 상품 등록, 변경 등 이벤트 시 1회이며, 1회 색인 시에는 질의 1건 227ms(GPU 185ms, 이후 1단계 완료 검증의 재실측은 238ms/193ms) 측정되어 속도 차이가 크지 않음을 확인하였다. 특히 임베딩 CPU 사용의 가장 큰 결정 이유는 범용 모델 변경 `Gemma4 12B` -> `DeepSeek R1 8B`으로 인한 추론 품질 손실과 임베딩을 VRAM으로 올렸을 때 트레이드 오프 비교 시 후자의 이득이 압도적으로 적다고 판단된다.

    추가로 현재 VRAM 가용 메모리도 약 2GB이므로 KV 캐시를 고려했을 때 리스크가 있는 상황이므로 로드맵 분석 평가 단계에서 실사용 벤치마크(다중 요청 시의 OOM 및 컨텍스트 고갈 여부 등) 후 리스크가 높다고 판단 시 모델 변경을 고려해야할 것으로 보인다.

## 2026-07-09

### 1. [Sprint 2, 게이트웨이 기반 구축] 테스트 전략 결정
* **맥락 및 배경 (Context):** FastApi를 통한 파이썬 서버 구축은 처음이므로 AI 도구에 구현을 위임하는 범위에서 구현 안정성 측면의 리스크가 있어 테스트 전략 결정이 필요하다. 설계 판단과 아키텍처 결정, 검증 기준 정의는 직접 담당한다.
* **결정 사항 (Decision):** 단위 기능 완성마다 기능의 구성 요소별 단위 테스트와 기능의 통합 테스트를 필수로 실시하고 테스트 실패 시 구조 재검증, 트렌드 파악을 실시하고 수정 방향에 대한 최종 컨펌 시 수정을 지침으로 작성하여 위임 구현에 의한 리스크를 최소화하는 것으로 결정했다.
* **트레이드 오프 (Trade-off):** 테스트 코드 작성으로 인한 개발 시간이 증가할 수 있으나 리스크는 초기에 수정하는 것이 총비용 관점에서 유리하다고 판단하여 해당 테스트 전략으로 결정하였다.

### 2. [Sprint 2, 게이트웨이 기반 구축] 게이트웨이 규격 결정
* **맥락 및 배경 (Context):** 현재 추론 서버의 게이트웨이(API) 규격을 Ollama의 기본 규격으로 갈지, OpenAI의 규격으로 갈지 결정이 필요하며 규격에 따라 향후  클라이언트 설정 방식이 달라진다. 
* **결정 사항 (Decision):** OpenAI 규격을 사용하는 것으로 결정하였다. 기본적으로 Ollama에서도 OpenAI의 규격을 지원하며, 리서치 시 OpenAI 규격이 업계 표준에 가깝다는 결과를 얻었기 때문이다.

    결정적으로는 현재 클라이언트(프로젝트 백엔드)의 AI 기능 테스트 규격을 OpenAI 설정 기준으로 작성하였기 때문에, 설정 최소 변경 차원에서 OpenAI 규격을 사용하는 것으로 결정하였다.
* **트레이드 오프 (Trade-off):** 추론 서버의 서빙 프레임워크로 Ollama를 사용하기 때문에 Ollama 독자 규격을 사용하는 것이 Ollama의 고유 기능 활용 관점에서는 구현이 더 간단할 수 있으나, 위에서 설명한 업계 표준, 클라이언트 변경 최소화에 맞춰 OpenAI를 사용하는 것으로 결정하였다. 

## 2026-07-10

### 1. [Sprint 3, 스트리밍 응답] 게이트웨이 요청 검증 책임 결정

* **맥락 및 배경 (Context):** 게이트웨이 규격은 OpenAI 규격을 사용하는 것으로 결정하였다. 이에 JSON 형식이 깨진 요청을 게이트웨이에서 직접 거절할지, 단순 중계 방식으로 Ollama까지 전달할지 결정이 필요하다. 스트리밍 응답은 게이트웨이가 요청 본문을 직접 확인해야 하므로 요청 검증 책임도 함께 명확히 해야 한다.

* **결정 사항 (Decision):** 클라이언트 요청 규격에 대한 검증은 게이트웨이가 담당하는 것으로 결정하였다. 따라서 JSON 파싱이 불가능한 요청은 Ollama로 전달하지 않고 게이트웨이에서 OpenAI 규격의 400 오류를 반환한다.

* **트레이드 오프 (Trade-off):** 게이트웨이가 요청 검증을 담당하면 단순 중계 방식보다 구현과 테스트 범위가 늘어난다. 하지만 클라이언트는 잘못된 요청도 일관된 OpenAI 규격의 오류 형식으로 받을 수 있다. 또한 모델 요청 방식에 대한 검증도 게이트웨이에서 처리할 예정이므로, 요청 검증 책임을 한 곳에 두는 것이 합리적이라 판단하였다.

### 1. [Sprint 4, 모델 라우팅과 임베딩] 추론 모델 선택 책임 위치 결정

* **맥락 및 배경 (Context):** 게이트웨이는 OpenAI 규격을 사용하기로 하였으나 OpenAI는 사용자(클라이언트)가 기능을 정의하고 모델은 사용자가 선택하는 방식이다. 하지만 현재 서버는 하드웨어 제약(vram 11GB)으로 기능별 모델을 하드웨어 상황에 맞게 최적화하여 결정한 구조이며 위의 방식 그대로 사용하는 것에는 제약이 있어 추론 모델 선택 방식에 대한 결정이 필요하다.

* **결정 사항 (Decision):** 클라이언트는 `chat`, `vision`, `embed`와 같은 별칭으로 기능별로 필요한 추상화된 추론 모델만 호출하며, 실제 기능별 사용 모델은 추론 서버가 선택하는 것으로 결정하였다.

* **트레이드 오프 (Trade-off):** 새로운 모델을 시험하거나 추가하려면 먼저 게이트웨이 설정에 별칭과 모델 매핑을 등록해야 하므로 직접 모델명을 전달하는 방식보다 자유도는 낮아진다. 하지만 제한된 하드웨어 환경에서는 기능별로 추상화된 별칭 방식을 사용하는 것이 성능과 메모리 효율 관점에서 유리하다고 판단하였다.

### 2. [Sprint 4, 모델 라우팅과 임베딩] 사고(thinking) 모드 동작 방식 결정

* **맥락 및 배경 (Context):** 범용 추론 모델로 채택된 `gemma4`는 사고 모드를 지원하며, ollama의 기본 응답이 사고 모드이므로 추론 품질을 우선한다. 그러나 사고 모드는 일반 모드 대비 지연 시간이 매우 길어 모든 경우에 적용하는 것은 비효율적이므로 사고 모드 동작 방식에 대한 결정이 필요하다.  

* **결정 사항 (Decision):** 요청에 대한 추론 모드는 기본적으로 일반 모드로 동작하고, 클라이언트가 추가로 사고 모드로 동작 요청 시 사고 모드로 동작하는 것으로 결정하였다.

* **트레이드 오프 (Trade-off):** 클라이언트마다 `reasoning_effort`를 요청에 포함하는 방식이 다르며 일부 클라이언트는 해당 필드를 직접 구성해야 할 수 있다. 하지만 게이트웨이는 OpenAI 표준 추론 깊이 필드를 그대로 사용하므로 특정 클라이언트 기술에 종속된 별도 계약을 추가하지 않아도 된다고 판단하였다.

### 1. [Sprint 5, 폴백] 생성 모델 폴백 정책 결정

* **맥락 및 배경 (Context):** 홈 서버의 로컬 추론이 중단될 경우 챗봇과 이미지 분석 기능도 함께 중단되므로, 외부 모델을 이용한 대체 추론 경로가 필요하다. 다만 이전 결정에 따라 게이트웨이가 클라이언트 요청 검증과 실제 모델 선택을 담당하므로, 어떤 기능과 오류를 폴백 대상으로 판단할지 함께 결정할 필요가 있다.

* **결정 사항 (Decision):** `chat`, `vision` 요청은 로컬 추론이 불가능할 경우 OpenAI의 `gpt-5-mini`로 폴백하는 것으로 결정하였다. 연결 실패, 응답 시간 초과, 서버 오류, 로컬 모델 사용 불가와 같이 정상적인 요청에 대해 로컬 서버가 추론 응답을 제공하지 못하는 경우만 폴백 대상으로 하며, JSON 파싱 실패나 요청 필드 검증 실패와 같은 클라이언트 오류는 게이트웨이에서 바로 반환한다.

  또한 로컬 장애가 반복될 경우 요청마다 동일한 실패와 대기 시간을 거치지 않도록 회로 차단기를 적용하고, 일정 시간이 지난 뒤 로컬 요청을 다시 시도하여 성공 시 자동으로 복구하는 것으로 결정하였다. 스트리밍 요청은 응답이 클라이언트에 전달되기 전에 실패한 경우만 폴백하며, 일부 응답이 전달된 이후에는 서로 다른 모델의 응답을 이어 붙이지 않고 오류로 종료한다.

* **트레이드 오프 (Trade-off):** 외부 모델 사용에 따른 API 비용과 로컬 모델과의 응답 차이가 발생할 수 있으며, 회로 차단기 상태와 스트리밍 전환 시점을 관리해야 하므로 구현 범위도 늘어난다. 하지만 로컬 장애가 지속되는 상황에서 매 요청마다 타임아웃을 반복하는 것보다 빠르게 대체 경로를 사용하고, 복구 이후 별도 조작 없이 다시 로컬 추론을 사용하는 것이 서비스 연속성 측면에서 더 유리하다고 판단하였다.

### 2. [Sprint 5, 폴백] 임베딩 프로세스 분리 및 복구 방식 결정

* **맥락 및 배경 (Context):** `chat`, `vision`은 요청 단위로 다른 모델을 사용할 수 있지만, 임베딩은 모델이 변경되면 기존 벡터와 벡터 공간이 달라져 검색 정합성이 깨진다. 따라서 임베딩은 외부 모델로 폴백할 수 없으며, 기존 `snowflake-arctic-embed2` 모델을 유지하면서 장애 영향을 줄일 수 있는 방식이 필요하다.

* **결정 사항 (Decision):** `snowflake-arctic-embed2`는 범용 추론 모델과 분리된 CPU 전용 Ollama 프로세스에서 실행하고, 프로세스 종료 시 동일한 설정과 모델로 자동 재기동하도록 구성하는 것으로 결정하였다. 게이트웨이는 `embed` 요청을 임베딩 전용 주소로 전달하며, 재기동 중에는 다른 모델로 우회하지 않고 OpenAI 규격의 오류를 반환한다.

* **트레이드 오프 (Trade-off):** 별도의 Ollama 인스턴스를 운영하므로 관리해야 할 프로세스와 실행 설정이 늘어난다. 하지만 임베딩은 CPU 실행 시에도 단건 요청 기준으로 현재 프로젝트의 요구 성능을 충족하며, 범용 추론 프로세스의 장애와 분리할 수 있다. 다른 모델로 폴백하여 기존 데이터 전체를 재색인하는 것보다 동일한 모델의 프로세스를 독립적으로 복구하는 것이 검색 데이터의 정합성과 자원 효율 측면에서 더 합리적이라 판단하였다.

### 1. [Sprint 6, 외부 노출과 인증] 추론 서버 외부 공개 방식 결정

* **맥락 및 배경 (Context):** 외부에 제공해야 하는 것은 추론 기능이며, Ollama와 내부 실행 환경까지 공개할 필요는 없다. 따라서 외부 진입점을 게이트웨이로 제한하면서 홈 네트워크의 노출을 최소화할 방법이 필요하다.

* **결정 사항 (Decision):** Cloudflare Named Tunnel을 통해 FastAPI 게이트웨이만 외부에 공개한다. 범용 추론과 임베딩 Ollama는 내부에서만 실행하며, 모든 외부 요청은 게이트웨이의 검증과 라우팅을 거치도록 구성한다.

* **트레이드 오프 (Trade-off):** 외부 접근이 Cloudflare와 인터넷 연결에 의존하지만, 공유기 포트를 직접 개방하는 것보다 내부 실행 환경을 보호하고 공개 경계를 통제하기에 적합하다고 판단하였다.

### 2. [Sprint 6, 외부 노출과 인증] 추론 API 인증 정책 결정

* **맥락 및 배경 (Context):** 추론 서버의 연산 자원은 한정적이기 때문에, 어떤 호출 주체에게 자원 사용을 허용하는지에 대한 인증 정책 결정이 필요하다.

* **결정 사항 (Decision):** 호출 주체별 API 키를 발급하고 게이트웨이에서 직접 검증한다. 키는 `Authorization: Bearer <API_KEY>` 형식으로 전달하며, 서버에는 원본 대신 해시값만 저장하여 개별 키의 폐기가 가능하도록 구성한다.

  이 방식은 OpenAI와 다른 한정 자원을 배분하는 API들의 인증 방식을 참고하여 결정하였다.

* **트레이드 오프 (Trade-off):** 여러 키를 관리해야 하는 부담은 생기지만, 실제 상용 LLM이라고 가정 시에 키 유출이나 권한 회수의 영향을 특정 호출 주체로 제한할 수 있으므로 공통 키보다 안전하다고 판단하였다.

### 3. [Sprint 6, 외부 노출과 인증] 추론 서버 장애 격리 및 복구 정책 결정

* **맥락 및 배경 (Context):** 이전 결정에서 생성 모델은 로컬 장애 시 외부 모델로 폴백하고, 임베딩은 다른 모델로 우회하지 않고 동일한 모델을 재기동하도록 결정하였다. 이 복구 전략이 동작하려면 게이트웨이와 각 추론 프로세스의 장애가 서로에게 전파되지 않도록 실행 구조를 분리해야 한다.

* **결정 사항 (Decision):** 게이트웨이, 추론 모델, 임베딩 모델, 외부 연결 터널을 서로 독립된 백그라운드 프로세스로 실행하고, 별도의 감시 프로세스가 장애 상태를 확인하여 복구 기능을 담당한다.

  이를 통해 범용 추론 장애 시에도 게이트웨이는 외부 모델로 폴백할 수 있고, 임베딩 장애는 동일한 모델 프로세스만 독립적으로 복구할 수 있도록 한다.

* **트레이드 오프 (Trade-off):** 프로세스별 실행 설정과 상태 관리 항목은 늘어나지만, 하나의 프로세스 장애가 전체 추론 서버 중단으로 이어지는 것을 방지하고 장애 영향을 해당 기능으로 제한할 수 있으므로 독립 실행 구조가 더 적합하다고 판단하였다.
