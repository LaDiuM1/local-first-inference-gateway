# 추론 서버 구현 로드맵 (ROADMAP)

> 이 문서는 파이널 프로젝트의 자체 추론 서버 구축을 위한 전체 단계와 진행 현황을 추적한다.
> 각 구현 단계는 사전 정의된 검증 기준을 통과해야 완료로 간주한다.
> 세부적인 기술적 의사결정 이력은 [`DECISIONS.md`](DECISIONS.md)에 있다.

## 진행 원칙

1. **단계별 검증 (Step-by-step Verification)** 
   - 각 단계는 이전 단계의 안정성을 기반으로 진행되므로, 반드시 검증 스크립트(`scripts/verify_stageN.py`)를 통과한 후 다음으로 넘어간다.
2. **실측 기반 의사결정 (Data-Driven Decisions)** 
   - 현재 제한된 하드웨어(GTX 1080 Ti) 환경을 고려하여, 메모리 적재량, 추론 속도, 응답 품질 등은 모두 실제 측정값을 기준으로 판단한다.
3. **인터페이스 일관성 (Interface Consistency)** 
   - 외부 클라이언트의 구현 기술과 관계없이 게이트웨이가 지원하는 엔드포인트의 요청·응답 계약은 OpenAI API 규격으로 통일한다. 지원 범위와 제약은 연동 가이드에 명시하고, 내부 모델 변경이나 처리 로직이 클라이언트에 누출되지 않도록 구조적으로 격리한다.

## 목표 아키텍처

```text
[openAt MSA 클라이언트 (사용 기술 독립)]
      │  POST /v1/chat/completions · /v1/embeddings (OpenAI 호환, model = 용도별 별칭)
      ▼
[외부 노출·인증 계층]  ※ 접근 방식·인증 방식 미결정 (6단계)
      ▼
[추론 게이트웨이 (FastAPI, 홈 PC)]  별칭 라우팅 · 스트리밍 중계   ※ 관측 범위 미결정 (7단계)
      ├─ 1차: [Ollama] chat·vision → gemma4:12b-it-qat (GPU, 11434) / embed → snowflake-arctic-embed2 (CPU 전용, 11435)
      └─ 폴백: chat·vision 로컬 장애 시 → OpenAI gpt-5-mini (embed는 폴백 제외)
```

구체적으로 표기한 구성은 1~5단계에서 확정·구현된 부분이고, `※ 미결정`으로 표기한 부분은 예정 단계(6~9)에서 결정한다.

**기술 스택:** Python 3.12+ / uv · FastAPI · httpx(async) · pydantic-settings + YAML · ruff · Ollama

## 단계별 상세 진행 사항

### 1단계. 모델 결정 - 모델 검증 [완료]
현재 구축된 하드웨어 환경(VRAM 11GB)의 제약을 고려하여 최적의 범용 추론 모델 및 임베딩 모델을 결정하고, 실제 구동 환경에서 성능을 검증한다.

* **작업 내용**
  * 하드웨어 한계 내에서 구동 가능한 Ollama 지원 모델들의 추론 성능 및 VRAM 점유율 리서치
  * 최종 후보군(`gemma4:12b`, `snowflake-arctic-embed2`) 다운로드 및 1차 양자화 테스트
  * 100% 가동 시의 VRAM 가용성, CPU/GPU 적재 분리, 토큰 생성 속도(TPS) 실측
* **실측 및 검증 결과 [26.07.08]**
  * **범용 추론 (`gemma4:12b-it-qat`):** 컨텍스트 8k 기준 GPU VRAM 7.3GiB 점유 확인. 생성 속도 약 30.7 tok/s 실측.
  * **임베딩 (`snowflake-arctic-embed2`):** VRAM 확보를 위해 CPU 적재로 우회. 1회 요청 시 238ms 측정(GPU 사용 시 193ms로 CPU 사용 시 속도 저하 미미함).
  * **최종:** 동시 상주 시 VRAM 약 1.2GiB 여유 확보. QAT 양자화 및 CPU 분리 전략이 유효함을 입증하여 챗봇/색인 요구사항을 모두 충족함.

### 2단계. 인프라 구축 - 게이트웨이 기반 구축 [완료]
OpenAI 호환 클라이언트가 표준 Base URL 설정으로 추론 서버에 접근할 수 있도록 API 인터페이스를 구축하고, OpenAI SDK를 기준으로 호환성을 우선 검증한다.

* **작업 내용**
  * `uv` 패키지 매니저를 활용한 프로젝트 초기화 및 포매팅(`ruff`) 환경 구성
  * FastAPI 기반 뼈대 앱 작성 (헬스체크 `/health`, 논스트리밍 `/v1/chat/completions` 엔드포인트 구현)
  * pytest 기반 계약 테스트 환경 구성 (인프로세스 ASGI 호출, 업스트림 Ollama는 목킹)
* **검증 기준**
  * [x] OpenAI SDK를 사용하는 클라이언트에서 Base URL만 변경하여 요청 성공 확인
* **실측 및 검증 결과 [26.07.09]**
  * **SDK 호환:** OpenAI SDK에서 Base URL만 게이트웨이로 변경하여 챗 요청·응답 수신 성공(응답 모델명·토큰 사용량 정상 수신). OpenAI SDK 기준으로 클라이언트 코드 수정 없는 전환이 성립함을 확인.
  * **투명 중계:** 게이트웨이는 요청·응답 본문에 개입하지 않음 — 존재하지 않는 모델 요청 시 업스트림(Ollama)의 404 에러가 OpenAI 에러 포맷 그대로 클라이언트 SDK에서 해석됨을 확인.
  * **품질 게이트:** 계약 테스트 8건(pytest)과 ruff 검사 통과. 실제 Ollama 대상 검증은 `scripts/verify_stage2.py`로 상시 재실행 가능.

### 3단계. 핵심 기능 - 스트리밍 [완료]
사용자가 응답 대기 시간을 길게 느끼지 않도록(TTFT 최적화), 전체 문장을 버퍼링하지 않고 생성되는 토큰 단위로 즉각 중계한다.

* **작업 내용**
  * `httpx async`를 활용하여 Server-Sent Events(SSE, `stream: true`) 중계 로직 구현
  * 형식이 깨진 요청은 게이트웨이가 OpenAI 규격 400으로 직접 거절 — 게이트웨이 요청 검증 책임의 시작(다음 단계의 모델 요청 검증으로 확장)
* **검증 기준**
  * [x] 클라이언트에서 응답을 토큰 단위로 순차 수신하는지 확인 (`gemma4`의 경우 내부 사고(thinking) 과정이 먼저 수신되므로, 전체 델타 데이터 기준으로 검증)
* **실측 및 검증 결과 [26.07.10]**
  * **순차 수신:** 델타 598개가 3.03초(첫 델타)~25.29초(마지막)에 시간축으로 분산 수신됨 — 전체 버퍼링이면 종료 시점에 몰리므로 토큰 단위 중계가 성립함을 확인.
  * **thinking 영향:** 첫 content 델타는 21.60초로 사고(thinking) 구간이 실제 답변 체감을 지연시킴을 실측. 사고 모드 동작 방식은 다음 단계(모델 라우팅과 임베딩)에서 함께 다룸.
  * **품질 게이트:** 계약 테스트 13건(pytest)과 ruff 검사 통과. 실제 Ollama 대상 검증은 `scripts/verify_stage3.py`로 상시 재실행 가능하며 2단계 스크립트 회귀 통과도 확인함.

### 4단계. 핵심 기능 - 모델 라우팅과 임베딩 [완료]
클라이언트의 모델 요청 방식과 실제 모델 선택 책임을 검토한 결과, 클라이언트는 `chat`, `vision`, `embed`와 같은 기능별 별칭만 호출하고 실제 모델의 매핑은 게이트웨이 설정 파일이 담당하는 것으로 결정하였다.

* **작업 내용**
  * YAML 라우팅 설정 파일(`routing.yaml`) 구성 및 `GATEWAY_` 설정 체계 연결 — 별칭·엔드포인트·실제 모델 매핑을 코드가 아닌 설정에서 로드
  * `/v1/chat/completions` 별칭 치환 중계 — `reasoning_effort`가 없으면 기본 일반 모드(`none`)를 넣고 있으면 그대로 전달하며, `model`만 실제 모델명으로 치환하고 나머지 필드는 stream 여부와 무관하게 그대로 전달
  * `/v1/embeddings` 엔드포인트 구현 — native `/api/embed`의 `options.num_gpu: 0`으로 CPU 실행을 강제하고 응답을 OpenAI embeddings 형식으로 변환(문자열·문자열 배열 입력, 순서·인덱스 보존, `encoding_format` float/base64 지원 — base64는 float32 리틀엔디언 바이트)
  * 엄격한 별칭 계약 — 실제 모델명·미등록 별칭·엔드포인트 불일치 별칭은 업스트림 호출 없이 OpenAI 규격 400으로 거절
  * 기동 시 라우팅 구성 오류(파일 누락·잘못된 YAML·중복/불완전 라우팅)를 런타임으로 미루지 않고 명확히 실패
  * 사고 모드는 기본 일반 모드 — 클라이언트가 OpenAI 표준 필드 `reasoning_effort`로 사고 수준을 지정하고, 비문자열·빈 문자열 `reasoning_effort`는 업스트림 호출 없이 400으로 거절(docs/DECISIONS.md 26.07.13 결정 2)
* **검증 기준**
  * [x] 서버 설정(Config) 변경만으로 요청이 처리되는 대상 모델이 즉시 전환되는지 확인
  * [x] 임베딩 API 호출 및 결과 반환 성공
* **실측 및 검증 결과 [26.07.13]**
  * **별칭 라우팅:** 클라이언트가 `chat`/`vision`/`embed` 별칭만 보내도 설정된 실제 모델로 치환돼 응답 수신. 실제 모델명(`gemma4:12b-it-qat`) 직접 지정·오타 별칭(`chatt`)·엔드포인트 불일치 별칭(chat 경로의 `embed`)은 업스트림 호출 없이 400으로 거절됨을 확인.
  * **사고 모드:** 기본(일반 모드, `reasoning_effort` 생략)에서 사고 델타 0개, `reasoning_effort: high`에서 사고 델타 398개로 관측 — 게이트웨이가 생략 시 `none`을 넣고 지정 값은 그대로 전달함을 실측. 비문자열·빈 문자열 `reasoning_effort`는 업스트림 호출 없이 400으로 거절됨을 확인.
  * **CPU 임베딩:** `embed` 별칭 단건 요청이 1024차원 벡터를 OpenAI 형식으로 반환하고(SDK 기본 base64 왕복), 배열 입력은 순서·index(`[0, 1]`)를 보존. `encoding_format: base64` 명시 시 SDK가 되돌릴 수 있는 float32 바이트를 반환. 요청 처리 후 임베딩 모델 VRAM 점유 0MiB로 native `num_gpu: 0`의 CPU 상주가 실제로 반영됨을 확인.
  * **품질 게이트:** 계약 테스트 78건(pytest)과 ruff 검사 통과. 실제 Ollama 대상 검증은 `scripts/verify_stage4.py`로 상시 재실행 가능하며 1~3단계 스크립트 회귀 통과도 확인함.

### 5단계. 핵심 기능 - 폴백 [완료]
홈 서버 특성상 로컬 추론이 중단되면 클라이언트 요청 전체가 실패하는 단일 장애점(SPOF)이 된다. 로컬 추론 장애 시에도 서비스 연속성을 확보하는 것이 이 단계의 목표다.

* **확정 범위 (docs/DECISIONS.md 2026-07-14)**
  * 폴백 대상 — `chat`·`vision` 별칭만 로컬 장애 시 OpenAI `gpt-5-mini`로 우회하고, `embed`는 벡터 공간 정합성을 위해 다른 모델로 폴백하지 않음
  * 폴백 조건 — 게이트웨이는 정상이나 로컬이 유효한 추론 응답을 주지 못하는 경우(연결 실패·시간 초과·429·5xx·모델 부재 404·응답 시작 전 빈/유효하지 않은 응답)만 폴백. 클라이언트 계약 오류와 모델 부재 외 로컬 4xx는 그대로 전달
  * 자동 복구 — `chat`·`vision` 별칭별 회로 차단기(연속 3회 실패 시 30초 open, 이후 half-open 단일 probe 성공으로 복구)
  * 임베딩 가용성 — chat·vision과 분리된 CPU 전용 임베딩 Ollama 인스턴스(11435)와 자동 재기동 감독
* **작업 내용**
  * 로컬 chat·임베딩·OpenAI 폴백 client의 책임과 수명 분리 — 각기 별도 base URL과 별도 `httpx.AsyncClient`를 lifespan이 소유
  * 폴백 자격 명시 — 외부로 우회하는 별칭 목록(`chat`·`vision`)을 한 곳(`FALLBACK_ELIGIBLE_ALIASES`)에만 두고, 이 목록으로만 회로 차단기를 만들어 데이터 반출 정책을 한눈에 감사 가능하게 함. 목록에 없는 chat 별칭은 로컬로만 라우팅하고 장애 시 OpenAI를 호출하지 않으며 OpenAI 규격 로컬 업스트림 오류를 반환. 폴백 대상 모델(`gpt-5-mini`)은 계약상 고정이라 배포 설정으로 다른 모델을 향하게 열지 않고 모듈 상수로 둠
  * 별칭별 회로 차단기 — HTTP 중계와 분리한 상태 기계로 구현하고, 주입한 monotonic clock으로 closed/open/half-open 전이와 half-open 단일 probe 동시성을 처리
  * `chat`·`vision` 폴백 중계 — 로컬 응답을 폴백 대상/비대상으로 구분하고, OpenAI로 `model` 치환과 `reasoning_effort` 변환(로컬 일반 모드 `none` → 저지연 `minimal`, 명시 값은 그대로) 후 상태·본문·스트림을 중계
  * 요청·JSON 경계 일원화 — chat·embedding 요청과 로컬·OpenAI 성공 응답이 작은 공용 strict JSON parser를 사용해 비표준 상수 `NaN`·`Infinity`뿐 아니라 파싱 결과가 무한대가 되는 `1e400`도 동일하게 거절함. chat의 `stream`은 생략 또는 boolean만 허용해 문자열·숫자·null 등은 어느 업스트림도 호출하기 전에 OpenAI 규격 400으로 반환함. 무효한 로컬 chat 2xx는 폴백 대상, 무효한 OpenAI 2xx는 비밀 없는 502로 처리하고, 유효 응답은 재직렬화 없이 application 바이트를 보존하며 폴백 대상 아닌 로컬 4xx는 본문이 비어 있어도 그대로 전달
  * 스트리밍 경계 — HTTPX의 전송 압축을 명시적으로 디코딩하고 `content-encoding`을 제거해 헤더와 중계 바이트를 일치시킨 뒤, 응답 시작 전 첫 완결 SSE `data` 이벤트(빈 줄로 종료된 `choices` 리스트)까지만 검사함. 검사 상한은 임의 transport 청크 전체가 아니라 명시적인 첫 이벤트 종료 오프셋까지 적용하고, 같은 큰 청크의 후속 이벤트는 복사 확대 없이 보존함. 전송 청크가 SSE 라인·이벤트나 CRLF를 쪼개도 처리하며 CRLF·CR·LF 줄 끝, 마지막 바이트가 CR인 CR-only 빈 줄, blank·keep-alive 주석과 `event`·`id`·`retry` 같은 data 외 SSE 필드를 지원함. 이벤트 종료 전 EOF·상한 초과·유효 이벤트 전 `[DONE]`은 커밋 전 무효로 전환하고, 확정 후에는 application 바이트를 손실·중복·재인코딩 없이 흘림. 첫 이벤트를 전달한 뒤의 전송 장애는 회로 차단기에 실패를 기록하고 예외를 다시 전파해 정상 EOF가 아닌 오류로 종료하며 provider를 섞지 않음. 비-2xx 스트림 본문을 읽는 중 연결이 끊겨도 응답을 한 번만 닫고, 로컬 비폴백 4xx는 상태를 보존한 일반 오류로 반환하며 OpenAI 오류 본문 실패는 비밀 없는 502로 합성함
  * HTTP 중계 헤더 — 정적 hop-by-hop 헤더뿐 아니라 `Connection` 헤더가 대소문자와 공백을 달리해 지명한 동적 헤더도 buffered·streaming 공통 필터에서 제거함
  * 비밀 비노출 — `OPENAI_API_KEY`는 표준 환경변수에서 `SecretStr`로만 읽어 오류·로그에 담지 않고, 키 미설정이나 OpenAI 연결 실패·무효 응답 시 비밀 없는 OpenAI 규격 502를 합성
  * `/v1/embeddings`는 임베딩 전용 client만 호출하고 OpenAI 폴백을 절대 호출하지 않으며, 정확히 native 200인 표준 JSON 본문만 성공 후보로 인정함. 응답 모델명이 요청한 실제 모델과 같은지, 요청 입력 수와 벡터 수가 일치하는지, 한 응답의 모든 벡터 차원이 같은지, 각 벡터가 float32로 표현 가능한 유한 수치의 비어 있지 않은 목록인지, `prompt_eval_count`가 있으면 bool이 아닌 0 이상 정수인지 확인하고 무효 본문과 200 외 2xx는 빈 성공 목록이나 직렬화 500이 아닌 비밀 없는 OpenAI 규격 502로 합성함. 4xx·5xx 전달과 유효 응답의 float/base64 인코딩, `num_gpu: 0` 동작은 그대로

자체 추론 서버가 실제 서비스에서 가지는 성능·품질·비용의 효용을 정량 데이터로 확인해야 한다. 비교 대상과 평가 기준을 세우고 그 결과를 산출하는 것이 이 단계의 목표다.

* **결정할 범위**
  * 평가 기준 — 어떤 성능·품질·비용 지표로 효용을 측정하고 무엇을 합격선으로 볼지
  * 비교 대상과 질의 셋 — 무엇과 비교하고 어떤 질문으로 품질을 평가할지
* **검증 기준**
  * [ ] 결정된 기준에 따라 로컬·비교 대상 간 효용 비교 결과표가 산출되는지 확인
  * [ ] 벤치마크 결과를 바탕으로 운영·라우팅 정책의 후속 조정 근거가 도출되는지 확인

## 범위 제외 (Non-Goals)

다음 항목들은 현재 스프린트 및 하드웨어 한계 상 이번 프로젝트 범위에서 명시적으로 제외한다.

* **vLLM / TGI 등 별도 서빙 엔진 도입:** 현재 보유한 GPU(GTX 1080 Ti)는 Pascal 아키텍처(Compute Capability 6.1)로 최신 서빙 프레임워크 지원 대상에서 제외됨
* **멀티 노드 분산 처리 및 K8s 기반 오토스케일링:** 단일 로컬 홈 서버 환경이므로 컨테이너 오케스트레이션 배제
* **자체 모델 파인튜닝 (Fine-tuning):** RAG(검색 증강 생성) 프롬프트 엔지니어링으로 도메인 지식을 해결하며, 모델 가중치 훈련은 진행하지 않음
