# 추론 서버 결정 이력 (Decision Log)

> 파이널에서 담당하는 자체 추론 서버의 핵심 아키텍처와 기술적 결정 사항들을 기록해 두는 공간입니다.  
> 각 결정의 배경 및 판단 근거와 트레이드 오프를 작성함으로써 결정 이력 추적 및 향후 비효율적 결정이라는 판단이 섰을 시 피드백 작성을 통해 개발 사고 확장을 위한 목적으로 사용됩니다.
>
> **배경:** 팀 프로젝트(openat) 파이널에 RAG 등 AI 활용 기능이 추가되며, 제한된 외부 API 토큰이 상위 모델 선택과 충분한 추론량까지 제약하면 기능 품질도 함께 낮아질 수 있다고 판단하였습니다. 이에 AI 서비스가 호출할 LLM 추론 백엔드를 홈 GPU 서버(GTX 1080 Ti)에 직접 구축합니다. 로컬 모델을 1차로 사용하고 장애 시 대체 추론 경로를 통해 서비스 연속성을 확보하는 하이브리드 구조를 검토합니다.

## 2026-07-01

### 1. 파이널 프로젝트 담당 기능 결정

* **맥락 및 배경 (Context):** 파이널 프로젝트에서 RAG와 이미지 분석 등 LLM 연동 기능을 추가하기 시작하였으나, 데브코스에서 지원하는 API 토큰은 팀이 사용할 수 있는 양이 제한되어 있었다. 이 제약은 단순히 호출 횟수를 줄이는 데서 끝나지 않고 상위 모델 선택과 충분한 추론량을 제한하여, 결과적으로 각 기능의 품질을 낮출 수 있다고 생각하였다. 이에 외부 토큰 예산이 프로젝트의 LLM 품질을 결정하는 구조 자체를 해결할 문제로 정의하였다.
* **결정 사항 (Decision):** LLM에 사용되는 추론 서버를 직접 구현하기로 결정하였다. 목적은 API 비용 자체를 줄이는 것이 아니라, 보유 장비로 반복 사용 가능한 추론량을 확보하고 프로젝트가 요구하는 품질을 유지할 수 있는 경로를 만드는 것이다. 최근 오픈소스 모델의 성능과 보유한 GTX 1080 Ti의 실행 가능성을 먼저 확인하였으며, 후속 벤치마크로 로컬 추론을 신뢰할 수 있는 범위를 검증하기로 하였다.
* **트레이드 오프 (Trade-off):** 제한된 하드웨어에서는 로컬 모델의 품질이 요구 수준에 미치지 못할 수 있고, 익숙하지 않은 Python과 FastAPI 기반 서버의 구축·운영 비용도 직접 부담해야 한다. 하지만 외부 토큰 제약에 맞추어 기능 품질을 낮추는 것보다 추론 경로를 직접 소유하고, 벤치마크 결과를 로컬 경로의 적용 범위와 후속 라우팅 정책을 정하는 근거로 사용하는 편이 프로젝트 목적에 더 적합하다고 판단하였다.

상기 사유가 이 프로젝트를 생성하게 된 결정 및 배경이다.

* **트레이드 오프 (Trade-off):** 세부 파라미터 제어나 고도화된 동시 처리 최적화에는 불리할 수 있다. 현재 환경에서는 실행 가능성이 우선이므로 Ollama를 채택하되, 이후 평가에서 성능이 부족하면 지원 범위 안에서 별도 최적화가 필요하다.

## 2026-07-08

### 1. [Sprint 1, 모델 결정] 추론 서버 하드웨어 환경에 최적화된 모델 결정

- **맥락 및 배경 (Context):** 현재 로컬 서버 하드웨어 환경에서 품질·추론 기능 요구를 고려하여 모델별 실측값을 기반으로 최적의 모델을 선정해야 한다.
- **결정 사항 (Decision):** 검색 임베딩은 `snowflake-ko` 후보 계열, 챗봇·이미지 분석은 `gemma4:12b`를 사용하는 것으로 결정하였다.
  - **선정 기준**

    현재 프로젝트의 추론 서버 요구는 챗봇·이미지 분석을 위한 범용 추론 모델과 검색 임베딩 모델 두 가지다.
    - `gemma4:12b`: 현재 하드웨어에서 사용 가능한 Ollama 범용 모델의 추론 능력을 비교했을 때 `Gemma4 12B`가 가장 높은 지표를 보였다. 1순위는 `gemma4:12b`로 정하되, 양자화 모델도 VRAM이 감당하지 못하면 `DeepSeek R1 8B` 등의 경량 후보를 다시 검토하기로 하였다.

      `gemma4 12b 8k`는 실측 시 VRAM 8.2GB와 유휴 점유 1.5GB를 포함해 총 11.248GB를 사용하여 여유가 부족했다. QAT 4비트 양자화 모델로 변경하자 가용 VRAM이 약 400MB 늘어 1.9GB의 여유를 확보했으며, 최종 모델 변경 여부는 임베딩 배치 전략과 후속 부하 검증을 함께 보고 판단하기로 하였다.
    - `snowflake-ko`: 범용 모델의 품질을 위해 VRAM을 우선 배정하면 임베딩 모델을 GPU에 함께 상주시키기 어렵다. 임베딩은 상품 등록·변경 이벤트에서 수행하므로 CPU 처리도 허용 가능한지 검토했고, 한국어 임베딩 리더보드와 CPU 자원을 기준으로 무거운 `pplx-embed-v1-4b`보다 상대적으로 가벼운 snowflake-arctic-embed 계열 후보를 선택하였다.
    - 후보 비교에는 [한국어 임베딩 리더보드](https://github.com/OnAnd0n/ko-embedding-leaderboard)를 참고하였다.

> **표기 정리:** `snowflake-ko`는 후보를 검토할 때 사용한 표기다. 해당 후보는 snowflake-arctic-embed 계열의 한국어 파인튜닝 파생판이지만 Ollama 공식 라이브러리에는 원본 다국어 모델만 있어, 1단계 실측부터 실제 Ollama 식별자는 `snowflake-arctic-embed2`를 사용하였다. 가벼운 임베딩을 CPU로 분리한다는 결정 취지와 실측 판단 기준은 같다.

- **트레이드 오프 (Trade-off):** 임베딩을 GPU에서 CPU로 옮기면 내부 토큰 배치 처리량은 21.7에서 3.4 청크/s로 약 6.5배 느려졌다. 다만 실제 시나리오의 단건 색인은 CPU 238ms, GPU 193ms로 차이가 작았다. 범용 모델을 더 작은 모델로 바꿀 때의 추론 품질 손실보다 임베딩을 GPU에 유지할 때의 이득이 작다고 판단해 CPU 분리를 선택하였다. 남은 VRAM은 약 2GB이므로 KV 캐시와 동시 요청의 OOM 위험은 후속 실사용 벤치마크에서 다시 확인하기로 하였다.

* **결정 사항 (Decision):** 외부 계약은 OpenAI 규격으로 통일하기로 결정하였다. Ollama가 해당 규격을 지원하고, 프로젝트 백엔드의 AI 기능도 OpenAI 설정을 기준으로 작성되어 있어 Base URL 변경만으로 전환 가능한 경계를 만들 수 있기 때문이다.
* **트레이드 오프 (Trade-off):** Ollama 고유 기능을 직접 노출하는 방식보다 게이트웨이의 변환·검증 책임이 늘 수 있다. 하지만 클라이언트를 특정 서빙 엔진에서 분리하고 변경 범위를 줄이는 이점이 더 크다고 판단하였다.

### 1. [Sprint 3, 스트리밍 응답] 게이트웨이 요청 검증 책임 결정

* **맥락 및 배경 (Context):** 게이트웨이 외부 계약은 OpenAI 규격으로 정하였다. 스트리밍 중계를 구현하려면 게이트웨이가 요청 본문을 직접 해석해야 하므로, 형식이 깨진 요청을 이 경계에서 거절할지 Ollama까지 전달할지도 함께 정해야 한다.

* **트레이드 오프 (Trade-off):** 단순 중계보다 구현과 테스트 범위는 늘어난다. 대신 클라이언트는 업스트림 구현과 무관하게 일관된 오류를 받고, 이후 모델 별칭과 옵션 검증도 같은 경계에서 처리할 수 있어 책임이 분산되지 않는다.

### 1. [Sprint 4, 모델 라우팅과 임베딩] 추론 모델 선택 책임 위치 결정

* **맥락 및 배경 (Context):** OpenAI 규격은 보통 클라이언트가 모델명을 직접 선택하지만, 11GB VRAM 환경에서는 기능별 모델 조합과 적재 위치를 서버가 통제해야 한다. 실제 모델명을 외부 계약에 노출하면 모델 교체 때마다 모든 클라이언트를 수정해야 한다.
* **결정 사항 (Decision):** 클라이언트는 `chat`, `vision`, `embed`처럼 기능을 나타내는 공개 별칭만 요청하고, 실제 모델과 업스트림 엔드포인트의 매핑은 게이트웨이 설정이 담당한다.
* **트레이드 오프 (Trade-off):** 새 모델을 시험하거나 추가할 때 게이트웨이 설정에 별칭과 매핑을 먼저 등록해야 하므로 클라이언트의 임의 선택 자유는 줄어든다. 대신 제한된 하드웨어에서 모델 적재와 교체 책임을 한 곳에 모으고 클라이언트를 내부 모델명에서 분리할 수 있다.

### 2. [Sprint 4, 모델 라우팅과 임베딩] 사고(thinking) 모드 동작 방식 결정

* **맥락 및 배경 (Context):** `gemma4`의 사고 모드는 답변 품질을 높일 수 있지만 일반 모드보다 첫 응답이 크게 늦어진다. 모든 요청에 기본 적용하면 대화형 기능의 체감 지연이 커지므로 호출자가 요청 성격에 따라 선택할 수 있어야 한다.
* **결정 사항 (Decision):** 기본은 일반 모드로 두고, 클라이언트가 `reasoning_effort`를 명시한 요청만 해당 사고 수준으로 전달한다. 게이트웨이는 OpenAI 호환 필드를 그대로 사용하되 잘못된 형식은 업스트림 호출 전에 거절한다.
* **트레이드 오프 (Trade-off):** 사고 모드를 사용할 클라이언트는 요청별 판단과 필드 구성을 직접 해야 한다. 하지만 별도 사설 계약을 만들지 않고 표준 필드 안에서 선택하게 하므로 특정 클라이언트 기술에 대한 결합은 늘지 않는다.

## 2026-07-14

### 1. [Sprint 5, 폴백] 생성 모델 폴백 정책 결정

* **맥락 및 배경 (Context):** 홈 서버의 로컬 추론이 중단될 경우 챗봇과 이미지 분석 기능도 함께 중단되므로, 외부 모델을 이용한 대체 추론 경로가 필요하다. 다만 이전 결정에 따라 게이트웨이가 클라이언트 요청 검증과 실제 모델 선택을 담당하므로, 어떤 기능과 오류를 폴백 대상으로 판단할지 함께 결정할 필요가 있다.
* **결정 사항 (Decision):** `chat`, `vision` 요청은 로컬 추론이 불가능할 경우 OpenAI의 `gpt-5-mini`로 폴백한다. 연결 실패, 응답 시간 초과, 서버 오류, 로컬 모델 사용 불가와 같이 정상적인 요청에 로컬 서버가 추론 응답을 제공하지 못하는 경우만 폴백 대상으로 하고, JSON 파싱 실패나 요청 필드 검증 실패 같은 클라이언트 오류는 게이트웨이에서 바로 반환한다.

  로컬 장애가 반복되면 요청마다 같은 실패와 대기 시간을 거치지 않도록 회로 차단기를 적용하고, 일정 시간 뒤 로컬 요청을 다시 시도하여 성공하면 자동 복구한다. 스트리밍 요청은 응답이 클라이언트에 전달되기 전에 실패한 경우만 폴백하며, 일부 응답을 전달한 뒤에는 서로 다른 모델의 응답을 이어 붙이지 않고 오류로 종료한다.
* **트레이드 오프 (Trade-off):** 외부 모델 사용에 따른 API 비용과 로컬 모델과의 응답 차이가 발생할 수 있고, 회로 차단기 상태와 스트리밍 전환 시점을 관리해야 하므로 구현 범위도 늘어난다. 하지만 로컬 장애가 지속될 때 매 요청마다 타임아웃을 반복하는 것보다 빠르게 대체 경로를 사용하고, 복구 뒤 별도 조작 없이 다시 로컬 추론을 사용하는 편이 서비스 연속성에 유리하다고 판단하였다.

* **트레이드 오프 (Trade-off):** LLM 심판에는 편향 가능성과 채점 비용이 따른다. 정답형 자동 채점을 우선하고 주관 평가만 블라인드 심판에 맡겨 영향을 제한한다. 동시 부하 측정도 시간이 들지만 이를 생략하면 결론이 단건 조건에만 성립하므로, 모델 결정 때 남긴 위험을 이 단계에서 확인하는 편이 합리적이다.

### 2. [Sprint 9, 벤치마크] 비교 대상과 질의 셋 결정

* **맥락 및 배경 (Context):** 생성 폴백은 `gpt-5-mini`, 검색 모듈 호환 계약은 `gpt-5.4-nano`와 `text-embedding-3-small`을 기준으로 한다. 벤치마크가 실제 운영 판단에 쓰이려면 비교 대상과 질의 셋도 이 구조가 답해야 하는 질문에 맞춰야 한다.
* **결정 사항 (Decision):** 세 모델을 모두 비교한다. `gpt-5-mini`는 장애 시 품질 변화와 로컬 기본 경로의 효용을, `gpt-5.4-nano`는 검색 기능 생성 경로 전환의 득실을, `text-embedding-3-small`은 임베딩 전환이 검색 품질을 지키는지를 답한다. 생성 비교는 같은 문항·출력 상한을 사용하고, 각 모델이 지원하는 저지연 일반 모드로 측정한다. 로컬은 일반 모드와 사고 모드를 함께 측정하되 합격 판정은 일반 모드로 한다.

  성능·부하·비용과 임베딩은 서비스 도메인을 가정한 한국어 자체 시나리오 셋으로, 생성 품질은 공개 객관 셋(KMMLU 서브셋)과 자체 고난도 셋으로 나눈다. 쉬운 과제의 고득점만으로 결론을 내리지 않고, 동일 문항에서 모델별 강점과 약점이 갈리는 지점까지 확인하기 위해서다.
* **트레이드 오프 (Trade-off):** 자체 셋은 외부 수치와 직접 비교할 수 없고 대표성이 설계 품질에 의존한다. 하지만 목적이 범용 리더보드 순위가 아니라 이 서비스에서의 효용 확인이므로 감수한다. 임베딩 비교까지 포함해 범위는 늘지만, 전체 재색인을 수반하는 전환의 품질 근거를 같은 평가에서 확보하는 편이 합리적이다.
