# 추론 서버 운영 가이드

이 문서는 외부 클라이언트 연동 계약이 아니라 홈 서버 운영 절차를 설명한다. 공개 API 사용자는 `/docs`의 연동 가이드만 보면 된다. 실제 토큰, API 키, 호스트명은 이 문서나 명령 기록에 남기지 않는다.

## 구성 원칙

Cloudflare의 원격 관리형 Named Tunnel은 외부 HTTPS 연결만 게이트웨이로 전달한다. 게이트웨이, 범용 Ollama, 임베딩 Ollama, Watchdog은 서로 독립된 Windows 시작 작업으로 실행하며 cloudflared는 공식 Windows 서비스로 실행한다. 공유기 포트포워딩은 만들지 않고 Ollama는 loopback 바인딩을 유지한다.

Watchdog은 공개 주소를 검사하지 않는다. 로컬 엔드포인트, 관리 대상 작업·서비스 이름, 작업 제어 스크립트와 로그 경로는 배포 코드에 고정되어 `.env`나 환경변수로 바꿀 수 없다. 연속 실패한 구성 요소 하나만 재시작하며, 게이트웨이 `/health`의 `401`은 운영 키 설정 오류로 기록하고 게이트웨이를 재시작하지 않는다.

## SYSTEM 실행 경로와 배포 사본

네 작업은 SYSTEM 최고 권한으로 실행된다. 이 권한으로 실행하는 코드를 비관리자가 고칠 수 있으면 로컬 쓰기 권한이 그대로 SYSTEM 코드 실행이 되므로, 작업은 개발 작업 트리를 실행하지 않는다. 설치 스크립트가 실행 대상을 관리자 전용 디렉터리 `%ProgramData%\local-first-inference-gateway\app`으로 복사하고 작업은 그 사본만 실행한다.

배포 사본은 관리자가 승인한 시점의 작업 트리 스냅숏이다. 작업 트리를 고쳐도 다시 배포하기 전까지는 SYSTEM 실행에 영향을 주지 않으며, 반대로 코드를 바꿨다면 다시 배포해야 반영된다. 사본에는 SYSTEM이 실행하거나 import하는 것만 담는다 — `src`, `scripts`, `.venv`, `routing.yaml`, `docs/API.md`와 게이트웨이 설정 `.env`(있을 때)다. 사본 디렉터리는 상속을 끊고 SYSTEM과 Administrators만 남기며 소유자도 Administrators로 두므로, 저장소의 `.env`보다 오히려 읽을 수 있는 주체가 좁다.

사본을 만들고 잠그는 것만으로는 격리가 성립하지 않는다. `uv sync`가 만드는 가상환경은 프로젝트를 editable로 설치하므로, 원본 작업 트리의 `src`를 Python 검색 경로에 얹는 포인터 파일이 `.venv` 안에 남는다. 이 파일을 그대로 복사하면 사본의 Python이 사본이 아니라 원본 트리의 코드를 읽는다. 그래서 설치 스크립트는 복사한 뒤 그 포인터를 사본 안의 `src`로 옮기고, 사본의 Python에게 실제 검색 경로와 `gateway` 패키지 위치를 직접 물어 배포 사본과 검증된 Python 배포본 밖을 가리키는 항목이 하나도 없는지 등록 전에 확인한다. 설정으로 코드나 공개 문서 경로를 열지 않으며, Watchdog 작업 제어 스크립트와 공개 `/docs` 원본은 자신이 속한 배포본 안에 고정한다.

사본은 배포 자리에서 직접 고치지 않는다. 네 작업은 이미 배포 자리를 가리키고 있어서, 그 자리를 먼저 갱신하고 나중에 검증하면 검증에 실패해도 검증하지 않은 코드가 그 자리에 남고 다음 작업 시작이나 재부팅에서 SYSTEM이 그대로 실행한다. 그래서 설치 스크립트는 사본을 `%ProgramData%\local-first-inference-gateway\app.staging`에 새로 만들어 ACL과 실제 import 경로를 모두 확인한 뒤에만 배포 자리로 옮기고, 옮긴 뒤의 확인이 실패하면 직전 배포본을 그대로 되돌린다.

교체하려면 배포 자리를 잡고 있는 프로세스가 없어야 한다. 네 작업은 모두 배포 사본을 작업 디렉터리로 쓰므로 하나라도 실행 중이면 Windows가 그 디렉터리를 옮기지 못한다. 그래서 설치 스크립트는 사본을 만들고 검증하는 동안에는 기존 서비스를 그대로 두고, 검증을 통과한 뒤에만 이 저장소가 소유한 실행 중 작업을 중지한다. Watchdog을 가장 먼저 중지한다 — 나중에 세우면 정비 중에 멈춘 작업을 도로 복구해 교체를 막는다. 제한 시간 안에 멈추지 않는 작업이 있으면 배포 자리를 건드리기 전에 멈추고 그때까지 중지한 작업만 원래대로 되돌린다. 이름만 같고 이 저장소가 등록하지 않은 작업은 조회도 중지도 하지 않으며, 배포 사본을 쓰지 않는 cloudflared 서비스는 교체를 이유로 중지하지 않는다.

교체 뒤의 확인, 운영 키 발급, 작업 등록, 작업 시작 중 어느 단계가 실패하든 새 배포본으로 올라온 작업을 남기지 않고 직전 배포본을 되돌린 다음, 갱신 전에 실행 중이던 작업만 다시 세운다. 되돌리는 것까지 실패하면 원래 원인과 복구가 실패한 이유를 함께 알리고 실패로 끝낸다 — 이때는 관리자가 배포 자리와 작업 상태를 직접 확인해야 한다.

설치 스크립트는 등록 전에 SYSTEM이 실행할 모든 경로가 승인된 관리 주체(SYSTEM, Administrators, TrustedInstaller) 외에는 수정 불가능한지 검증한다. 대상은 작업 action의 실행 파일, Ollama 실행 파일, Python 배포본과 표준 라이브러리, 배포 사본 전체다. 소유자는 권한 설정과 무관하게 권한을 다시 쓸 수 있으므로 소유자도 함께 본다. 개별 사용자 계정에 부여된 권한은 그 계정이 관리자이더라도 승인하지 않는다 — 권한 상승 없이 실행되는 프로세스가 그대로 쓸 수 있기 때문이다.

운영 상태 파일에도 같은 기준을 적용한다. `%ProgramData%`는 표준 사용자도 하위 디렉터리와 파일을 만들 수 있고, 디렉터리의 ACL을 다시 써도 이미 그 안에 있던 파일의 소유자와 명시적 권한은 그대로 남는다. 상태 디렉터리를 잠그는 것만으로는 미리 놓인 API 키 저장소나 Watchdog 보호 키가 보호되지 않는다는 뜻이다. 그래서 설치 스크립트는 상태 디렉터리를 먼저 잠가 검사 중 새 파일이 끼어들 틈을 없앤 뒤, 비관리자도 파일을 만들 수 있던 자리에서 발견된 파일은 누가 썼는지 확인할 수 없으므로 채택하거나 고치지 않고 그 자리에서 멈춘다. 관리 주체만 쓸 수 있던 자리의 파일과 새로 발급한 파일은 소유자를 Administrators로 되돌리고 명시적 권한을 지워 상태 디렉터리의 제한만 상속하게 한다 — 파일을 만든 관리자 계정이 소유자로 남으면 그 계정의 권한 상승 없는 프로세스가 소유자 권한으로 권한을 다시 써 machine-scoped DPAPI로 보호한 운영 키를 읽을 수 있기 때문이다.

인증 근거를 어디서 읽을지도 설정으로 열지 않는다. 게이트웨이가 읽는 API 키 저장소는 `%ProgramData%\local-first-inference-gateway\api-keys.json`, Watchdog이 읽는 보호 키는 `%ProgramData%\local-first-inference-gateway\watchdog-key.dpapi`로 코드에 고정되어 있고 환경변수나 `.env`로 바꿀 수 없다. 배포 사본의 `.env`는 작업 트리에서 복사되므로 비관리자가 내용을 정할 수 있는데, 이 자리가 설정이면 그 한 줄이 설치 스크립트가 잠근 경계 밖의 파일을 인증 근거로 만들어 아무 키나 통과시킨다. 설치 스크립트도 잠글 자리를 따로 적지 않고 배포할 코드에게 직접 물으므로, 잠그는 자리와 SYSTEM이 읽는 자리는 어긋날 수 없다.

`%ProgramData%`가 어디인지도 환경변수에서 읽지 않는다. 환경 블록은 주체마다 다르고 사용자가 자기 몫을 직접 정할 수 있는데, 설치 스크립트는 관리자 셸의 환경에서 실행되고 네 작업은 SYSTEM의 환경에서 실행된다. 이 자리를 환경에서 읽으면 두 환경이 다를 때 ACL을 적용한 자리와 SYSTEM이 읽는 자리가 어긋나고, 잠기지 않은 자리에 비관리자가 미리 놓아 둔 저장소를 그대로 인증 근거로 채택하게 된다. 그래서 관리자만 바꿀 수 있는 등록 정보를 보는 Windows 셸 API에 공용 데이터 폴더를 직접 물어, 어느 환경에서 물어도 같은 자리가 나오게 한다.

**따라서 Ollama와 Python은 관리자만 쓸 수 있는 위치에 설치해야 한다.** 사용자 프로필 아래에 설치하는 기본 옵션(`%LOCALAPPDATA%\Programs\Ollama`)이나 사용자별 Python 배포본은 이 조건을 만족하지 않는다. Ollama는 모든 사용자용으로 설치하고, 가상환경은 `C:\Program Files` 등에 설치한 Python으로 만든다. Ollama 공식 설치 프로그램은 사용자 프로필에만 설치하므로, 관리자 전용 위치에는 같은 버전의 standalone 아카이브를 풀어 둔다 — 이 저장소의 작업은 `ollama serve`만 실행하므로 CLI 아카이브로 충분하고, 자동 업데이터가 검증된 버전을 임의로 올리지도 않는다. 검증에 실패하면 스크립트는 어느 작업도 등록하지 않고 어떤 경로의 어떤 주체가 문제인지 출력하며 멈춘다. 이 스크립트는 직접 만든 디렉터리 밖의 권한을 바꾸지 않는다 — 작업 트리나 기존 설치의 ACL을 임의로 고치지 않으므로, 위치를 옮기거나 권한을 조정하는 것은 관리자가 판단해서 한다.

Ollama 데스크톱 앱의 자동 시작도 함께 끈다. 시작 프로그램에 등록된 트레이 앱은 로그인할 때마다 사용자 권한으로 `ollama serve`를 띄워 chat 포트를 먼저 차지한다. 그러면 이 저장소의 chat 작업은 포트를 잡지 못하고 게이트웨이의 chat 요청이 관리 밖 프로세스로 가므로, SYSTEM 실행 경로 격리가 chat에서만 무너지고 실행 주체가 로그인 여부에 따라 달라진다. 시작 프로그램 폴더의 Ollama 바로 가기를 지우거나 옮기면 chat 포트는 언제나 이 저장소가 등록한 작업이 잡는다.

## 최초 설치

1. 프로젝트 의존성을 동기화하고 `scripts/install_stage6.ps1 -ValidateOnly`를 실행한다. 이 모드는 시스템 상태를 바꾸지 않고 필수 파일, Python 환경, Ollama 실행 파일과 모델 저장소, 기존 작업과 cloudflared 서비스 유무를 확인하며, SYSTEM이 실행할 경로마다 `SAFE` 또는 `UNSAFE`를 보고한다. `UNSAFE`가 있으면 설치 전에 먼저 해결한다. OpenAI 폴백을 쓰려면 이 시점에 저장소 `.env`에 `OPENAI_API_KEY`를 둔다 — 없으면 게이트웨이는 로컬 전용으로 동작하고 폴백이 필요한 요청은 502로 끝난다.
2. Cloudflare 대시보드에서 기존 구성을 먼저 확인한 뒤 원격 관리형 Named Tunnel과 공개 호스트 하나를 구성한다. Origin은 게이트웨이 작업이 사용하는 loopback 주소 `http://127.0.0.1:29000`만 가리키고 Ollama 주소는 등록하지 않는다. 기존 Tunnel이나 DNS 레코드는 삭제하지 않는다.
3. Cloudflare 대시보드가 제공하는 `cloudflared.exe service install <TUNNEL_TOKEN>` 명령을 관리자 CMD에서 직접 실행한다. 토큰을 저장소 파일이나 문서, 채팅, 스크립트 인자 기본값에 옮기지 않는다. 이 절차는 Cloudflare의 [원격 관리형 Tunnel 설치 문서](https://developers.cloudflare.com/tunnel/setup/)를 기준으로 한다.
4. 관리자 PowerShell에서 `scripts/install_stage6.ps1 -StartTasks`를 실행한다. 스크립트는 상태 디렉터리와 배포 사본의 ACL을 SYSTEM과 관리자 그룹으로 제한하고, 배포 사본을 만든 뒤 Watchdog 운영 키를 원문 출력 없이 Windows DPAPI로 보호한다. 작업이 오른 뒤 라우팅이 요구하는 두 모델을 내려받는다 — `ollama pull gemma4:12b-it-qat`와 `ollama pull snowflake-arctic-embed2`. pull은 실행 중인 서버의 모델 저장소로 저장되므로 관리 chat 서버(11434)가 뜬 이 시점에 실행해야 두 작업이 공유하는 지정 저장소로 들어간다 — 데스크톱 Ollama 등 다른 서버가 떠 있으면 다른 저장소로 갈 수 있다. 설치 검증은 저장소 디렉터리의 존재만 확인하므로, `ollama ls`로 두 모델이 보이는지 여기서 확인한다.
5. `scripts/show_stage6_status.ps1`로 네 작업과 cloudflared 서비스 상태를 확인한다.

등록 스크립트는 같은 이름만 보고 기존 작업을 덮어쓰지 않는다. 저장소 소유 설명, 실행 파일, 인자가 모두 일치하는 작업만 갱신하며 다른 작업이 있으면 키 발급이나 배포를 시작하기 전에 중단한다. 제거 또는 기존 Tunnel 삭제 기능은 제공하지 않는다.

## 배포 사본 갱신

코드나 의존성을 바꾼 뒤에는 같은 명령을 다시 실행해 배포 사본을 갱신한다. 갱신은 매번 사본을 새로 만들어 검증한 뒤 교체하므로 반복 실행해도 결과가 같다.

```powershell
scripts/install_stage6.ps1 -StartTasks
```

실행 중인 작업은 미리 멈춰 둘 필요가 없다. 갱신이 교체 직전에 스스로 중지하고 교체를 마친 뒤 다시 세우므로, 중단 구간은 사본 검증이 끝난 시점부터 교체가 끝나는 시점까지다. `-StartTasks`를 주면 네 작업을 모두 세우고, 주지 않으면 갱신 전에 실행 중이던 작업만 되돌린다 — 일부러 멈춰 둔 작업은 갱신을 이유로 시작하지 않는다. 갱신은 사본을 매번 새로 만들므로 이전 배포에만 있던 파일은 사본에 남지 않는다 — 남은 옛 모듈이 import되는 일을 막기 위해서다. 갱신할 때마다 Python 검색 경로 포인터를 사본 안으로 다시 맞추고 실제 import 경로를 다시 확인하므로, 원본 트리에서 `uv sync`를 다시 실행해도 사본이 원본을 읽게 되지 않는다.

Cloudflare 캐시는 origin의 `Cache-Control`을 존중하도록 둔다. 기존에 모든 경로를 강제로 캐시하는 규칙이 있다면 그 규칙을 삭제하지 말고 `/v1/*`와 `/health`만 bypass 대상으로 추가한다. 6단계에서는 Cloudflare WAF 요청 제한이나 게이트웨이 키별 rate limit 규칙을 추가하지 않는다.

## 서비스 API 키 관리

API 키 관리는 관리자 PowerShell의 로컬 CLI에서만 수행한다. 발급 명령은 원본 키를 한 번 출력하므로 즉시 대상 서비스의 비밀 저장소로 옮긴다. 이 명령들은 게이트웨이가 읽는 보호된 저장소를 그대로 쓴다 — 다른 자리를 쓰려면 관리자가 `--store`로 명시해야 하고, 그렇게 만든 저장소는 운영 작업이 읽지 않는다.

발급과 폐기는 저장소를 새 파일로 만들어 바꿔친다. 그대로 두면 저장소 소유자가 명령을 실행한 개별 관리자 계정이 되고, 소유자는 권한 설정과 무관하게 권한을 다시 쓸 수 있으므로 그 계정의 권한 상승 없는 프로세스가 저장소를 열어 키를 심을 수 있다. 그래서 보호된 저장소에 쓸 때는 바꿔치기 전에 소유자를 Administrators로 옮기고, 옮기지 못하면 저장소를 바꾸지 않고 실패한다 — 설치 때만이 아니라 키를 관리할 때마다 소유자와 권한이 유지된다.

```powershell
.\.venv\Scripts\python.exe -m gateway.key_cli issue --client <service-name>
```

목록에는 키 식별자, 서비스 이름, 생성·폐기 시각만 나오며 원본이나 해시는 나오지 않는다.

```powershell
.\.venv\Scripts\python.exe -m gateway.key_cli list
```

개별 키는 목록의 비밀이 아닌 식별자로 폐기한다.

```powershell
.\.venv\Scripts\python.exe -m gateway.key_cli revoke --key-id <key-id>
```

MSA마다 별도 키를 발급하고 공통 키를 공유하지 않는다. 회전할 때는 새 키를 먼저 배포하고 호출 성공을 확인한 뒤 이전 키만 폐기한다.

## Watchdog 운영 키 복구

Watchdog이 게이트웨이에서 `401`을 받으면 대상 작업을 재시작하지 않는다. 관리자 PowerShell에서 키 목록과 작업 상태를 확인한 뒤 운영 키가 폐기됐거나 보호 파일과 불일치한다고 확인된 경우에만 다음 명시적 복구를 실행한다.

```powershell
scripts/install_stage6.ps1 -ReissueWatchdogKey -StartTasks
```

이 명령은 배포 사본을 교체한 뒤 새 운영 키를 발급해 DPAPI 보호 파일을 바꾼다. Watchdog은 보호 키를 기동 시 한 번만 읽으므로, 교체를 위해 이미 멈춰 있는 Watchdog이 새 파일을 쓴 뒤에 올라와 반드시 새 키를 읽는 인스턴스가 된다. 발급이 실패하면 직전 배포본과 갱신 전 실행 상태를 되돌린다 — 보호 파일로 옮기지 못한 키는 발급한 자리에서 곧바로 폐기하므로 쓸 수 없는 키가 남지 않는다.

이전 키는 교체 즉시 폐기되지 않으므로 복구 도중에도 인증이 끊기지 않는다. 목록에서 이전 `stage6-watchdog` 키를 확인한 뒤 사용하지 않는 키만 개별 폐기한다.

## 상태 확인과 장애 대응

`scripts/show_stage6_status.ps1`은 비밀을 읽지 않고 작업 상태, 마지막 실행 시각, 마지막 결과 코드와 cloudflared 서비스 상태만 보여 준다. 마지막 결과가 성공이 아니면 Windows 작업 스케줄러의 해당 작업 실행 기록에서 시작·종료 실패 원인을 확인한다. Watchdog의 비밀 없는 대상별 실패·재시작·설정 오류 기록은 관리자만 읽을 수 있는 전용 상태 디렉터리의 회전 로그에 남는다. 특정 로컬 구성 요소가 응답하지 않으면 Watchdog은 기본 3회 연속 실패 후 그 작업만 재시작하고, 반복 장애에는 1분부터 최대 10분까지 증가하는 cooldown을 적용한다.

게이트웨이는 모든 요청의 처리 경과를 상태 디렉터리 하위 `logs\requests.jsonl`(JSON Lines, 16MiB × 5개 회전)에 기록하고, 회로 차단기 상태 전환도 같은 로그에 이벤트로 남긴다. 클라이언트가 문의와 함께 전달한 `x-request-id`로 해당 줄을 찾으면 단계별 상대 시간, 최종 provider, 로컬 실패 사유를 대조할 수 있다. 프롬프트·응답 본문·API 키 원문은 기록되지 않으며, 로그 디렉터리는 상태 디렉터리의 권한을 상속해 관리자만 읽는다. 이 디렉터리를 열 수 없으면 게이트웨이는 기동에 실패하므로, 작업 트리에서 직접 실행하는 개발·검증 게이트웨이는 `GATEWAY_REQUEST_LOG_DIRECTORY`로 임시 경로를 지정해 운영 로그와 격리한다.

인터넷 또는 Cloudflare 장애 중에도 로컬 엔드포인트가 정상이면 게이트웨이와 Ollama 작업을 재시작하지 않는다. cloudflared는 서비스가 중지된 경우에만 시작하며 외부 연결 실패만으로 정상 서비스를 재시작하지 않는다.

외부망 호출과 로그인 없는 재부팅 복구는 실환경에서 확인했다. 결과는 `docs/ROADMAP.md`의 6단계에 있다.

## 운영 모니터

요청 관측 로그의 후처리와 스택 실시간 상태를 한 화면으로 보는 온디맨드 대시보드다. 관측 기록 결정과 같은 이유로 상주 수집 프로세스를 추가하지 않는다 — 필요할 때 실행하고 닫으면 남는 것이 없으므로 별도 제거 절차도 없다.

```powershell
scripts/run_monitor.ps1
```

모니터는 `127.0.0.1:29100`에만 바인딩되며 Cloudflare 터널이나 공개 호스트에 등록하지 않는다. 브라우저로 `http://127.0.0.1:29100/`을 열면 대시보드가 나오고, 같은 내용의 JSON은 `/api/status`로 받는다. 화면은 구성요소 생존(게이트웨이·두 Ollama·예약 작업 4종·cloudflared·GPU·디스크)과 관측 로그 기반 요청 지표(1시간·24시간 창의 요청 수, 운영 오류율, provider 분포, 응답 시작·전체 지연 p50/p95, 별칭별 분해, 회로 이벤트, 최대 동시 처리)를 보여 준다.

게이트웨이 생존은 키 없이 `/health`의 `401` 응답으로 판정하므로 모니터는 어떤 비밀도 읽지 않는다. 운영 관측 로그는 관리자만 읽을 수 있으므로 요청 지표까지 보려면 관리자 PowerShell에서 실행한다 — 비관리자 실행은 실시간 프로브만 표시된다. 개발·검증 로그를 보려면 `.venv\Scripts\python.exe -m gateway.monitor --log-dir <경로>`로 임시 경로를 지정한다. 결정적 검증은 `scripts/verify_monitor.py`로 상시 재실행할 수 있다.
