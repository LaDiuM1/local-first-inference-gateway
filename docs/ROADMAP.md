# 추론 서버 구현 로드맵 (ROADMAP)

> 이 문서는 파이널 프로젝트의 자체 추론 서버 구축을 위한 단계와 검증 계획을 추적한다.
> 각 단계는 사전에 정한 검증 기준을 통과해야 완료로 간주하며, 구체적인 기술 선택은 해당 단계의 실측과 결정 이력에 따라 확정한다.
> 세부적인 기술적 의사결정 이력은 [`DECISIONS.md`](DECISIONS.md)에 기록한다.

## 진행 원칙

1. **단계별 검증 (Step-by-step Verification)**
   - 각 단계는 이전 단계의 안정성을 전제로 하므로 검증을 마친 뒤 다음 단계로 넘어간다.
2. **실측 기반 의사결정 (Data-Driven Decisions)**
   - GTX 1080 Ti 11GB 환경에서 메모리 적재량, 추론 속도, 응답 품질을 직접 측정해 기술 선택을 확정한다.
3. **클라이언트와 추론 세부사항의 분리**
   - 외부 클라이언트가 내부 모델과 서빙 방식에 직접 의존하지 않도록 게이트웨이를 경계로 둔다. 구체적인 API 규격과 라우팅 책임은 구현 단계에서 비교해 결정한다.

## 초기 목표 구조

```text
[openAt MSA 클라이언트]
      │
      ▼
[추론 게이트웨이: 외부 계약·검증·중계]
      │
      ▼
[로컬 모델 서빙: 생성·이미지 분석·임베딩]
```

외부 공개, 인증, 장애 시 대체 경로는 홈 서버의 보안과 서비스 연속성을 함께 검토한 뒤 후속 단계에서 확정한다.

## 단계별 계획

### 1단계. 모델 검증

* 생성·이미지 분석·임베딩 후보를 실제 하드웨어에 적재한다.
* VRAM·RAM 점유, 생성 속도와 기본 응답 품질을 비교해 이후 구현에 사용할 후보를 정한다.

### 2단계. 게이트웨이 기반 구축

* 클라이언트 요청을 받아 로컬 서빙 엔진으로 중계하는 최소 API를 구현한다.
* 클라이언트 변경 범위와 서빙 엔진 종속성을 비교해 외부 계약을 결정한다.

### 3단계. 스트리밍 응답

* 생성 결과를 전체 버퍼링하지 않고 순차 중계한다.
* 첫 응답과 마지막 응답이 시간축으로 분산되는지 실측한다.

### 4단계. 모델 라우팅 및 임베딩

* 기능별 요청과 실제 모델의 매핑 책임을 정한다.
* 생성·이미지 분석·임베딩 경로를 분리하고 임베딩 결과의 순서와 차원을 검증한다.

### 5단계. 폴백 처리

* 로컬 장애와 클라이언트 오류를 구분해 대체 경로가 필요한 범위를 결정한다.
* 장애 시 연속성과 정상화 후 복귀 과정을 검증한다.

### 6단계. 외부 노출 및 인증

* 홈 네트워크의 직접 포트 개방을 피하면서 안정적인 외부 접근 지점을 구성한다.
* 인증되지 않은 요청이 추론 경로에 도달하기 전에 차단되는지 확인한다.

### 7단계. 시스템 관측

* 지연, 처리량, 대기와 장애 경로를 요청 단위로 재구성할 수 있게 한다.
* 프롬프트, 응답 본문과 인증 비밀은 관측 기록에서 제외한다.

### 8단계. MSA 클라이언트 연동

* 실제 MSA 기능 하나를 첫 연동 대상으로 정한다.
* 연동 절차, 계약 차이와 롤백 조건을 확인한다.

### 9단계. 벤치마크

* 로컬 경로의 단건·동시 요청 성능, 생성·임베딩 품질과 비용을 측정한다.
* 비교 대상과 합격 기준을 측정 전에 확정하고, 결과를 적용 범위와 후속 라우팅 정책의 근거로 사용한다.
